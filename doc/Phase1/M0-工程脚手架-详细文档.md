## 3. 动手前必须懂的 5 个概念

### 概念 1:TypeScript 是什么

**一句话:TypeScript = JavaScript + 类型标注**。它不是新语言,是 JS 的超集。你写的 `.ts` 文件,最终会被「编译/转译」成普通 `.js` 再运行。

类比你的 Vue 经验:你在 Vue 里写过 `defineProps({ count: Number })` 给 prop 标类型吧?TS 就是把这种「标类型」用到**所有变量、函数参数、返回值**上,而且是在**编译时**就帮你查错。

```typescript
// JS:谁知道 name 该是什么?传错了运行时才崩
function greet(name) {
  return 'hi ' + name.toUpperCase();
}

// TS:标注 name 是 string。传数字?编辑器当场标红
function greet(name: string): string {
  return 'hi ' + name.toUpperCase();
}
```

好处:**写错当场知道**(编辑器红线)、**自动补全特别准**(它知道每个对象有哪些字段)。这对你这种新手反而是福音——它会拉着你别犯错。

### 概念 2:Node.js 与 pnpm(包管理器)

- **Node.js**:让 JS 能脱离浏览器、在你电脑上直接运行的运行时。它能读写文件、开子进程——浏览器里的 JS 干不了这些。我们的插件后台逻辑就跑在 Node 里。
- **pnpm**:装第三方库的工具,跟你可能见过的 `npm` 一个角色,但更快、对「monorepo」支持更好。`package.json` 记录依赖,`pnpm install` 把它们下载进 `node_modules`。

类比:你建 Vue 项目时跑过 `npm create vue@latest`、`npm install` 吧?`pnpm` 就是更好用的 `npm`。

### 概念 3:Monorepo / workspace(一个仓库,多个包)

我们的项目拆成 4 个**包(package)**:`shared`、`extension`、`webview`、`agent-core`。它们住在**同一个 git 仓库**里,这叫 **monorepo**。

`pnpm-workspace.yaml` 告诉 pnpm「这些子目录各自是一个包」。好处:`extension` 可以像用第三方库一样 `import` 本地的 `shared` 包,pnpm 自动把它们「链接」起来,不用发布到网上。

类比:把它想成一个「项目群」,公共的类型放 `shared`,谁都能用。

### 概念 4:打包器 esbuild

你写的代码分散在很多 `.ts` 文件、还依赖 `node_modules` 里的库。**打包器**把这一大堆揉成一个能直接跑的 `.js` 文件。

类比:你用 Vue 时,背后是 **Vite** 在打包(Vite 内部其实就用了 esbuild)。区别是这次**我们自己直接配 esbuild**,因为 VS Code 插件要的产物格式很特定。

> 🔑 **一个最容易让新手懵的点**:`esbuild` 打包时**为了快,会直接把类型标注删掉、并不检查类型对不对**。所以我们另外用 TS 自带的 `tsc` 命令**专门做类型检查**。记住这个分工:
>
> - `pnpm build`(esbuild)= 出能跑的文件,**不查类型**
> - `pnpm typecheck`(tsc)= **只查类型**,不出文件
>
> 两个都要跑,各管一摊。

### 概念 5:VS Code 插件长什么样

一个 VS Code 插件有两块,跑在**两个不同环境**,这点务必分清:

|                               | 跑在哪                  | 能干啥                             | 我们的包                   |
| ----------------------------- | ----------------------- | ---------------------------------- | -------------------------- |
| **扩展主体 (extension host)** | Node.js 环境            | 读写文件、开子进程、调 LLM         | `extension` + `agent-core` |
| **Webview(聊天界面)**         | 类似浏览器的沙箱 iframe | 只做界面(React),**不能**直接读文件 | `webview`                  |

两者**不能直接互相调用**,只能用「发消息」(`postMessage`)通信——这就是为什么 `shared` 包里要定义「消息协议」类型。

- 插件的 `package.json` 不只是依赖清单,还是**插件清单**:声明这个插件提供哪些命令、什么时候被激活、入口文件是哪个。
- 入口文件导出两个函数:`activate()`(插件启动时调用)和 `deactivate()`(关闭时)。

> M0 里 webview 只占位,等 M1 才真正写 React 聊天界面。本阶段你只会碰到 `extension` 和 `shared`。

### 步骤 1 · 初始化仓库 + workspace 骨架

先初始化 git 并创建最外层文件。

```powershell
git init
```

**文件:`pnpm-workspace.yaml`**

```yaml
# 声明:packages/ 下每个子目录都是一个独立的包
packages:
  - 'packages/*'
```

**文件:`package.json`**(根目录的,注意它是"管理者",`private: true` 表示自己不发布)

```json
{
  "name": "embed-agent-monorepo",
  "version": "0.0.0",
  "private": true,
  "scripts": {
    "build": "node esbuild.mjs",
    "watch": "node esbuild.mjs --watch",
    "typecheck": "tsc --noEmit -p tsconfig.json",
    "test": "vitest run",
    "lint": "eslint \"packages/**/*.{ts,tsx}\"",
    "format": "prettier --write ."
  }
}
```

> `scripts` 里的每一项,都能用 `pnpm <名字>` 运行(比如 `pnpm build`)。现在还不能跑,因为工具和文件都没建。

**文件:`.gitignore`**

```gitignore
node_modules/
dist/
out/
coverage/
*.vsix
*.db
.DS_Store
```

**文件:`tsconfig.base.json`**(TS 的公共编译选项,各包都会继承它)

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "noEmit": true,
    "lib": ["ES2022", "DOM"]
  }
}
```

逐条人话解释:

- `target: ES2022`:把代码编译成 2022 版的 JS 语法(够新,Node 22 完全支持)。
- `strict: true`:**最严格模式**。对新手是好事——它逼你把类型写清楚,少踩坑。
- `moduleResolution: bundler`:告诉 TS「我用打包器」,这样它能正确找到我们 `shared` 包。
- `noEmit: true`:tsc **只检查、不生成文件**(生成的活交给 esbuild)。

**文件:`tsconfig.json`**(根目录,用于「一次性检查所有包」)

```json
{
  "extends": "./tsconfig.base.json",
  "include": ["packages/*/src"]
}
```

现在装根目录的工具依赖(`-D` = 开发依赖,`-w` = 装到 workspace 根):

```powershell
pnpm add -D -w typescript esbuild vitest prettier eslint @eslint/js typescript-eslint @types/node
```

**运行 → 你应该看到**:

```powershell
pnpm install
```

无报错,根目录出现 `node_modules/` 和 `pnpm-lock.yaml`。

### 步骤 2 · `shared` 包(顺便学 TS 类型)

这是本阶段唯一「有内容」的包。它只放**类型**和**一个小函数**,是你学 TS 的最佳样本。

**文件:`packages/shared/package.json`**

```json
{
  "name": "@embed-agent/shared",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "exports": {
    ".": "./src/index.ts"
  }
}
```

> `exports` 指向 `.ts` 源码——这叫「内部包」写法,别的包 `import` 它时直接读源码,**省去单独编译 shared 的步骤**。`@embed-agent/shared` 是它的包名,别人就用这个名字 import。

**文件:`packages/shared/tsconfig.json`**

```json
{
  "extends": "../../tsconfig.base.json",
  "include": ["src"]
}
```

**文件:`packages/shared/src/index.ts`**(重点!逐段都有注释,配合附录 A 看)

```typescript
// ========== 1) 工具返回契约(见 Phase1 计划 §五)==========
// interface = 描述"一个对象长什么样"。类比 Vue 里给 props 定义形状。
export interface Source {
  file: string; // 必填:来源文件路径
  lines?: string; // 问号 = 可选字段,比如 "1-40"
  section?: string; // 可选:文档小节标题
}

// <T> 是"泛型":调用时再决定 result 的具体类型(附录 A 有讲)。
// = unknown 是默认值:不指定时当作"未知类型"。
export interface ToolResult<T = unknown> {
  result: T;
  sources?: Source[]; // Source[] = "Source 组成的数组"
}

// ========== 2) 配置 ==========
//
// 用户在 VS Code 设置里填的东西。注意:API key 不在这里 —— key 太敏感,
// 走 VS Code 的 SecretStorage 存(见步骤 3 package.json 旁的说明),不写进普通配置。
//
// provider = 用哪家大模型。这里用"字面量联合"(只能填列出的这几个之一):
// 好处是填错当场报红;以后加新 provider 时,凡是没处理它的代码编译器都会提醒。
//   - 'anthropic':开发期默认,工具调用最准(见规划 §3.3)
//   - 'openai'   :备选
//   - 'deepseek' :便宜选项。它的接口"兼容 OpenAI",所以底层复用 OpenAI SDK,
//                  只要把下面的 baseURL 指到 https://api.deepseek.com 即可
export type LlmProvider = 'anthropic' | 'openai' | 'deepseek';

export interface AgentConfig {
  // 用哪家(上面三者之一)
  provider: LlmProvider;
  // 具体模型名,如 'claude-opus-4-7'、'deepseek-v4-flash'。取值开放 → string。
  model: string;
  // 自定义 API 端点(可选)。留空 = 用该 provider 的官方默认地址;
  // 填了就打到这个地址 —— DeepSeek、自建网关、任何"OpenAI 兼容"服务都靠它。
  // 例:用 DeepSeek 时填 'https://api.deepseek.com'。
  baseURL?: string;
}

// ========== 3) RPC 消息协议(Webview 与 Extension 互发的消息)==========
//
// 回忆概念 5:webview(聊天界面)跑在沙箱里,不能直接调用 extension(Node 后台)
// 的函数,两边只能靠 postMessage "发消息"。所以这里把"允许互发的消息"全部列成类型,
// 两个方向各一个类型,前后端共用 —— 谁都不会发错字段、也不会漏处理某种消息。
//
// 下面每个成员都是"可辨识联合"(discriminated union):靠 type 字段当"标签",
// 收到后用 switch (msg.type) 就能精确知道这条消息还带了哪些字段(附录 A 有详解)。
// 这是本项目最重要的 TS 模式,务必理解。

// ---- 方向一:Webview(界面)──► Extension(后台)----
export type WebviewToExt =
  // 用户在输入框敲了一句并发送 —— 一轮对话的起点,后台收到后启动 agent 循环。
  // text:用户输入的原文。
  | { type: 'userMessage'; text: string }
  // 用户点了"停止":要求中断当前正在流式输出的这轮回答。无额外字段。
  | { type: 'cancelStream' }
  // 用户对某条"确认请求"(见下方 requestConfirm)的回应。
  // id:对应是哪一条请求(可能同时挂着多条);approved:同意 / 拒绝 ——
  // 这就是"执行命令"等受控操作的"放行开关"(对应"write-heavy is gated"原则)。
  | { type: 'confirmResponse'; id: string; approved: boolean };

// ---- 方向二:Extension(后台)──► Webview(界面)----
export type ExtToWebview =
  // 回答的一小片增量文本(流式输出,一片一片来);界面把 text 追加到当前气泡。
  // 为什么不整段一次发?因为"必须流式",边生成边显示体验才好(见规划 §3.3)。
  | { type: 'streamDelta'; text: string }
  // agent 开始调用某个工具了;name 是工具名(如 'read_file')。
  // 界面可据此显示"正在查 xxx…",让用户看见 agent 在做什么(可审计)。
  | { type: 'toolCallStart'; name: string }
  // 某次工具调用结束;name 是哪个工具,ok 表示成功与否。界面更新那条状态。
  | { type: 'toolCallResult'; name: string; ok: boolean }
  // 本轮回答彻底结束(流式输出完了)。无字段;界面可借此重新启用输入框。
  | { type: 'assistantDone' }
  // agent 想执行一个需要用户点头的操作(如执行一条命令),请界面弹确认框。
  // id:之后用户的 confirmResponse 靠它对应回来;
  // command:将要执行的完整命令,原样给用户看清楚再决定(不允许偷偷执行)。
  | { type: 'requestConfirm'; id: string; command: string }
  // 出错了(LLM 报 429 / 超时、配置缺失等);message 是人类可读的说明。
  | { type: 'error'; message: string }
  // 本轮 / 累计 token 用量;input、output 分别是输入、输出 token 数。
  // 界面显示出来,让用户对"花了多少"心里有数(见规划 §3.3 token 计费)。
  | { type: 'tokenUsage'; input: number; output: number };

// ========== 4) 一个小工具函数(纯粹为了有东西可测试)==========
// 参数和返回值都标了类型。反引号是"模板字符串",你在 JS 里见过。
export function formatTokenUsage(input: number, output: number): string {
  return `${input} in / ${output} out`;
}
```

**文件:`packages/shared/src/index.test.ts`**(测试。`describe/it/expect` 是 vitest 提供的)

```typescript

import { formatTokenUsage } from './index';

describe('formatTokenUsage', () => {
  it('把输入输出 token 拼成一行', () => {
    expect(formatTokenUsage(10, 20)).toBe('10 in / 20 out');
  });
});
```

**运行 → 你应该看到**:


输出 `1 passed`(绿色)。🎉 你已经有一个会编译、能测试的 TS 包了。

### 步骤 3 · `extension` 包 + esbuild(出第一个真插件)

**文件:`packages/extension/package.json`**(这是「插件清单」,字段含义见注释下方)

```json
{
  "name": "embed-agent",
  "displayName": "Embed AI Agent",
  "description": "Conversational AI agent (VS Code extension)",
  "version": "0.0.1",
  "private": true,
  "engines": {
    "vscode": "^1.85.0"
  },
  "categories": ["Other"],
  "main": "./dist/extension.js",
  "activationEvents": [],
  "contributes": {
    "commands": [
      {
        "command": "embed-agent.hello",
        "title": "Embed Agent: Hello"
      }
    ]
  },
  "dependencies": {
    "@embed-agent/shared": "workspace:*"
  }
}
```

关键字段:

- `engines.vscode`:要求 VS Code ≥ 1.85(规划定的)。
- `main`:插件入口,指向 esbuild 打出来的 `dist/extension.js`(还没生成,马上)。
- `contributes.commands`:声明一个命令,会出现在命令面板里。
- `"@embed-agent/shared": "workspace:*"`:依赖**本地** shared 包(`workspace:*` = 用工作区里的版本)。

**文件:`packages/extension/tsconfig.json`**

```json
{
  "extends": "../../tsconfig.base.json",
  "include": ["src"]
}
```

装插件需要的「VS Code API 类型」(只装到 extension 这个包):


**文件:`packages/extension/src/extension.ts`**(插件入口)

```typescript
// 'vscode' 是宿主提供的 API,不需要安装运行时,只装了类型(@types/vscode)。
import * as vscode from 'vscode';
// 从本地共享包导入——这一行能跑通,就证明 monorepo 链接成功了。
import { formatTokenUsage } from '@embed-agent/shared';

// 插件被激活时,VS Code 调用这个函数。
export function activate(context: vscode.ExtensionContext): void {
  console.log('Embed Agent 已激活');

  // 注册命令:命令 id 必须和 package.json 里声明的一致。
  const disposable = vscode.commands.registerCommand('embed-agent.hello', () => {
    vscode.window.showInformationMessage(`Embed Agent 已就绪!(token: ${formatTokenUsage(0, 0)})`);
  });

  // 把命令登记到 subscriptions,插件卸载时自动清理。
  context.subscriptions.push(disposable);
}

// 插件关闭时调用,M0 暂时不需要做事。
export function deactivate(): void {}
```

**文件:`esbuild.mjs`**(打包脚本。`.mjs` 后缀表示它本身用 ES Module 语法)

```javascript
import esbuild from 'esbuild';

// 是否带 --watch 参数(改代码自动重新打包)
const watch = process.argv.includes('--watch');

const ctx = await esbuild.context({
  entryPoints: ['packages/extension/src/extension.ts'], // 从入口出发
  outfile: 'packages/extension/dist/extension.js', // 打成这一个文件
  bundle: true, // 把依赖都揉进来
  platform: 'node', // 目标是 Node 环境(不是浏览器)
  format: 'cjs', // VS Code 插件要求 CommonJS 格式
  target: 'node18',
  external: ['vscode'], // ⚠️ vscode 由宿主提供,绝不能打包进去
  sourcemap: true,
});

if (watch) {
  await ctx.watch();
  console.log('esbuild: 监听中,改动会自动重打包…');
} else {
  await ctx.rebuild();
  await ctx.dispose();
  console.log('esbuild: 打包完成 -> packages/extension/dist/extension.js');
}
```

> 注意 `external: ['vscode']`:`vscode` 这个模块运行时由 VS Code 注入,**打包进去反而会出错**——这是插件开发最经典的一条规则。

**文件:`.vscode/launch.json`**(让你按 F5 就能调试插件)

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "运行插件",
      "type": "extensionHost",
      "request": "launch",
      "args": ["--extensionDevelopmentPath=${workspaceFolder}/packages/extension"],
      "outFiles": ["${workspaceFolder}/packages/extension/dist/**/*.js"]
    }
  ]
}
```

**运行 → 你应该看到**:

```powershell
pnpm build
```

生成 `packages/extension/dist/extension.js`。然后:

1. 在 VS Code 里**按 F5**(或菜单「运行 → 启动调试」)。

### 步骤 6 · 持续集成(CI)

让 GitHub 在每次提交时,自动在 Windows/macOS/Linux 三个系统上跑一遍检查。

**文件:`.github/workflows/ci.yml`**

```yaml
name: CI
on: [push, pull_request]

jobs:
  check:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck
      - run: pnpm lint
      - run: pnpm test
      - run: pnpm build
```

> 这一步要等你把仓库推到 GitHub 后才会真正运行。本地无需操作。`--frozen-lockfile` 表示「严格按 lock 文件装,不许擅自改版本」,保证三台机器装的一模一样。

## 附录 A · 现代 JS / TS 速查(给 Vue + 简单 JS 背景的你)

下面这些语法你会在本项目反复见到,按需查。

| 语法                    | 长这样                                              | 人话 / 类比                                             |
| ----------------------- | --------------------------------------------------- | ------------------------------------------------------- |
| **类型标注**            | `let n: number = 1`                                 | 给变量贴个「类型标签」                                  |
| **interface**           | `interface User { id: number }`                     | 描述对象的形状;类似 Vue props 的类型定义                |
| **type 别名**           | `type Id = string \| number`                        | 给类型起名字,可表达「或」                               |
| **联合类型 `\|`**       | `'v1' \| 'v2'`                                      | 「只能是其中之一」                                      |
| **可选字段 `?`**        | `lines?: string`                                    | 这个字段可有可无                                        |
| **数组类型**            | `string[]`                                          | 一串字符串                                              |
| **泛型 `<T>`**          | `ToolResult<string>`                                | 「占位类型」,用时再填;类比函数参数,只不过填的是「类型」 |
| **import / export**     | `export function f(){}` / `import { f } from './x'` | 模块化;你在 Vue SFC 里见过                              |
| **default vs 具名导出** | `export default x` / `export const x`               | 默认导出 import 时不带 `{}`;具名要带 `{}`               |
| **箭头函数**            | `() => {}`                                          | 你熟                                                    |
| **async / await**       | `await fetch(...)`                                  | 等异步结果;比 `.then()` 链好读                          |
| **Promise**             | `Promise<string>`                                   | 「将来会有的一个 string」                               |
| **解构**                | `const { provider } = config`                       | 从对象里取字段;Vue 里常用                               |
| **展开 `...`**          | `[...a, ...b]`                                      | 摊开数组/对象                                           |
| **可选链 `?.`**         | `a?.b`                                              | a 为空就不往下取,不报错                                 |
| **空值合并 `??`**       | `a ?? 'default'`                                    | a 是 null/undefined 才用默认值                          |
| **模板字符串**          | `` `hi ${name}` ``                                  | 你熟                                                    |

**重点理解:可辨识联合(discriminated union)**——本项目消息协议的核心模式。

```typescript
type Msg = { type: 'text'; content: string } | { type: 'error'; code: number };

function handle(msg: Msg) {
  if (msg.type === 'text') {
    // 这个分支里,TS 自动知道 msg 有 content(没有 code)
    console.log(msg.content);
  } else {
    // 这里 TS 知道是 error,有 code
    console.log(msg.code);
  }
}
```

好处:你 `switch (msg.type)` 时,**编辑器会按当前分支精确提示有哪些字段**,几乎不可能取错字段。这就是为什么 `shared` 用它定义所有消息。

## 附录 B · 常见报错与排查

| 现象                                           | 多半原因                           | 处理                                                                    |
| ---------------------------------------------- | ---------------------------------- | ----------------------------------------------------------------------- |
| `Cannot find module '@embed-agent/shared'`     | 改了包后没重新链接                 | 根目录跑 `pnpm install`                                                 |
| `pnpm: 无法将...识别为命令`                    | pnpm 没装好                        | 回第 4 节,用 corepack 启用                                              |
| typecheck 报 `'x' is possibly 'undefined'`     | strict 模式发现你没处理空值        | 用 `?.`、`??`,或先判空                                                  |
| 按 F5 没反应 / 命令找不到                      | 没先 `pnpm build`,或命令 id 不一致 | 先 `pnpm build`;核对 `package.json` 里命令 id 与 `registerCommand` 一致 |
| 打包后插件运行报 `Cannot find module 'vscode'` | esbuild 没把 vscode 设为 external  | 检查 `esbuild.mjs` 里 `external: ['vscode']`                            |
| lint 报一堆格式问题                            | 没格式化                           | 先 `pnpm format` 再 `pnpm lint`                                         |

## 附录 C · 名词表

- **monorepo**:一个 git 仓库里放多个包。
- **package(包)**:一个有自己 `package.json` 的代码单元。
- **workspace**:pnpm 把 monorepo 里的多个本地包统一管理、互相链接的机制。
- **bundler(打包器)**:把多文件+依赖揉成一个可运行文件的工具,这里是 esbuild。
- **transpile(转译)**:把 TS 变成 JS。
- **extension host**:VS Code 里跑插件后台逻辑的 Node 进程。
- **webview**:插件里显示自定义界面的沙箱(本项目放 React 聊天界面)。
- **CI(持续集成)**:每次提交自动跑检查的流水线(这里用 GitHub Actions)。
