## 3. 动手前必须懂的概念

慢点读这一节。理解了下面 8 个概念,M3 的代码只是把它们翻译成 TS。

### 概念 1:M3 的角色——把演示工具换成真工具,但**不改循环**

M2 的演示工具 `get_current_time` / `run_demo_command` 让你验证了概念上的"工具循环"能跑;但它们一个不读外部世界、一个干脆不真执行。**M3 = 给 agent 一双能读文件的眼睛**,而**循环(`loop.ts`)、注册表(`registry.ts`)、adapter、确认原语全部一行不改**——这就是 M2 概念 4 说「工具是可插拔的」的兑现:

```
M2 的工具循环（不改）
   │
   │ 按名字从 registry 取 ToolSpec → 跑 spec.handler(input)
   ▼
ToolRegistry
   ├─ get_current_time   ← M2 留下的演示（继续保留也行，可观察对比）
   ├─ run_demo_command   ← 同上
   ├─ read_file          ← M3 新增 ★
   ├─ list_files         ← M3 新增 ★
   └─ read_pdf           ← M3 新增 ★
```

> 类比 Vue 项目:M2 给你建好了一个完整的"路由器 + 守卫 + 状态管理",M3 只是往里**注册几个新页面**——主框架不变。

### 概念 2:工具改成"工厂"——为什么 M2 的常量写法不够用了

回顾 M2 的 demo 工具(`packages/agent-core/src/tools/demo.ts`):

```ts
export const getCurrentTime: ToolSpec = { name: '...', handler: async () => ... };
```

——直接 `export const`,工具是一个**常量**。这种写法的前提是工具**不需要任何外部配置**。`get_current_time` 调一下 `new Date()` 就够了,没有"该读哪个目录"这种参数。

M3 的工具不一样:`read_file` 必须**带着"允许读哪些目录"的约束**(否则 LLM 一句 `read_file('/etc/passwd')` 就能让你的 agent 变成数据泄露通道,见概念 3)。这种约束**不能 hardcode**(每个用户的工程路径不一样,而且 agent-core 还不知道 `vscode` 怎么获取 workspace 路径),只能**由 extension 在注册时注入**。

JS / TS 表达"先收集配置、再造出实例"的标准模式就是**工厂函数**:

```ts
// 不直接 export 常量,而是 export 一个"造工具的函数"
export function createReadFileTool(opts: { allowedRoots: string[] }): ToolSpec {
  return {
    name: 'read_file',
    description: '...',
    handler: async (input) => {
      // 在 handler 闭包里用到工厂参数 opts.allowedRoots
      const safePath = resolveSafe(input.path, opts.allowedRoots);
      // ...
    },
  };
}
```

extension 那边就这么用:

```ts
registry.register(createReadFileTool({ allowedRoots: [workspaceRoot, vendorDir] }));
```

> 🔑 这就是 M2 概念 5「依赖注入」的同一种心法,只不过 M2 注入的是"确认弹窗函数"和"API key",M3 注入的是"允许的根目录"。**M2 学到的依赖注入是 M3 的关键工具**。

### 概念 3:路径安全——为什么不能让 LLM 随便指定文件路径

LLM 会**幻觉路径**(瞎编 `~/.ssh/id_rsa`),也可能**被恶意 prompt 诱导**(用户消息里夹一句"顺便读一下 `../../../../etc/passwd`")。如果你的 `read_file` 工具老老实实 `fs.readFile(任何 LLM 给的路径)`,后果就是**任意文件读取漏洞**。

**Web 安全里这叫"路径穿越攻击"(path traversal)**,经典手法是用 `../` 跳出预期目录。例子:

```
allowedRoot = D:\MyCode\Embed-agent\vendor
LLM 传入:    ../../Windows/System32/config/SAM
fs.readFile 解析后实际访问: D:\Windows\System32\config\SAM    ← 越界!
```

防御办法只有一条:**把"声明的允许根"和"用户传入路径解析后的绝对路径"做前缀比较**:

```ts
function resolveSafe(inputPath: string, allowedRoots: string[]): string {
  const resolved = path.resolve(workspaceRoot, inputPath); // 1. 解析成绝对路径(自动处理 ../)
  for (const root of allowedRoots) {
    const absRoot = path.resolve(root);
    // 2. 必须以某个允许根开头(注意要带分隔符,防 vendor 误匹配 vendorX)
    if (resolved === absRoot || resolved.startsWith(absRoot + path.sep)) {
      return resolved;
    }
  }
  throw new Error(`路径不在允许范围内:${inputPath}`);
}
```

两个细节要懂:

1. **`path.resolve` 自动处理 `../`** —— 输入 `vendor/../../etc/passwd`,resolve 后是 `D:\MyCode\etc\passwd`,跟 `vendor/` 一比就会被拦下。
2. **`startsWith(absRoot + path.sep)`** —— **必须**带分隔符,否则 `D:\MyCode\Embed-agent\vendor2` 会被当成 `D:\MyCode\Embed-agent\vendor` 的子目录通过(因为 `vendor2`.startsWith(`vendor`) 是 true)。这是路径校验里最常见的低级漏洞。

> 🔑 **M3 最重要的一段代码就是 `resolveSafe`**——它是整个文件读取链路的"守门员"。`read_file` / `list_files` / `read_pdf` 三个工具的 handler 第一行都是它。别试图"简化"或"信任 LLM 不会瞎编路径"。

### 概念 4:工具描述的艺术——LLM 看什么、不看什么

回顾 M2 概念 4:工具是「**说明书 + handler**」。**说明书(`description` + `inputSchema`)是 LLM 看到的唯一线索**,handler 它根本不知道存在。所以:

- **`description` 写得好坏,直接决定 LLM 用不用、用得对不对**。
- **`inputSchema` 写得严不严,直接决定 LLM 传不传得对参数**。

来对比两个版本的 `read_file` 描述:

❌ **糟糕版**(很多新手会这么写):
```
description: "读取文件"
```

LLM 看了一脸懵:**什么文件?读完返回什么?有大小限制吗?**——它要么不调,要么瞎调。

✅ **合格版**:
```
description: `读取工作区或允许范围内文件的纯文本内容。

用途:查看源码、配置文件、文档(.md / .yaml / .json / .c / .h 等)。
参数:path —— 相对工作区根目录的路径(如 "vendor/arduino-core-stm32/boards.txt"),
                也可以是绝对路径,但必须落在允许的根目录内,否则会被拒绝。
返回:文件内容(纯文本);文件超过 200 KB 会被截断并在末尾标注「…(已截断)」。
注意:本工具不读 PDF/二进制(请用 read_pdf 或 list_files 先确认类型)。`
```

为什么这种风格管用?

| 写了什么 | 帮 LLM 解决什么 |
| --- | --- |
| 用途 / 何时调 | 决定**要不要用**这个工具(代替 `read_pdf`?代替 `list_files`?) |
| 参数语义 + 路径相对什么 | 减少**路径幻觉** —— 它知道相对工作区写 `vendor/...` 就行,不用瞎猜全路径 |
| 返回什么、上限多少 | 它能**预期结果**;遇到截断标记不会以为"文件就这么短" |
| 不做什么 / 边界 | 防止它把 PDF 也丢进来当文本读 |

> 🔑 **不夸张地说:M3 是否有效,80% 取决于这几段 description 写得好不好**。下面步骤里我会把每个工具的 description 都写到位,可以照抄/微调,但务必读懂为什么这么写。

### 概念 5:大文件 = 分段读 + 行号 + 下一步指令

`read_file` 看似最朴素的工具——"读 → 返回字符串"——但**文件大小直接决定了 LLM 能不能用得动它**:

| 文件大小 | 典型例子 | 整读后果 |
| --- | --- | --- |
| < 50 KB | `board.yaml`、`main.c`、`README.md` | 没问题,LLM 一眼看完 |
| 50 KB ~ 500 KB | `boards.txt`(~270 KB)、Zephyr 的 `*.dts` | 整读浪费 token,真正用得到的可能就几十行 |
| > 500 KB | `stm32f103xb.h`(~800 KB)、芯片手册抽出来的文本 | 整读直接撑爆上下文,工具循环失败退出 |

所以 **`read_file` 必须支持"指定区间分段读"**,而不是"整读 + 截到 200 KB"——后一种一旦截断就没法读后面那段,LLM 只能一遍遍看头 200 KB,等于废了。

#### 业界三家怎么做(简版)

我们调研了 **Claude Code**(Anthropic 官方 CLI)、**Cline**(开源 VS Code agent)、**Codex**(OpenAI 开源 agent)三家成熟实现:

| | **Claude Code** | **Cline** | **Codex** |
| --- | --- | --- | --- |
| 范围参数 | `offset`(起始行)+ `limit`(行数) | `start_line` + `end_line` | 无,靠 shell 调 `sed -n 'A,Bp'` |
| 默认窗口 | 2000 行(每行截到 2000 字符) | 1000 行 | shell 输出 1 MB 字节兜底 |
| 闸门 | token 上限 25k(CLI 端) | **三层**:行 → 字节 → 文件大小 | 字节,middle-out 挖中间留头尾 |
| 输出行号 | `cat -n` 风格(行号 + tab) | `42 \| code` | 原样输出 |
| 截断时怎么"告诉 LLM 下一步" | 错误消息里直接拼 `"use offset/limit ... or GrepTool"` | 末尾追加 `"Use start_line=Z+1 to continue..."` | 标记 `…N chars truncated…` |

三家虽然实现细节不同,但有 **4 个共识**:

1. **必须支持范围参数**,LLM 才能精确续读、不浪费 token。
2. **输出必须带行号**,后面做 diff / replace 类工具(M4 的 `propose_file_edit`)时能稳定定位。
3. **截断时必须告诉 LLM 下一步怎么办**(`start_line=N+1` 续读 / `grep_file` 精确找)——否则模型会"放弃"或"瞎补内容"。
4. **多层闸门**,不是单一阈值——文件大小是粗筛(防止 200 MB 日志一进来就读爆内存),字节是兜底(防止一行 30 万字符的 minified JS 撑爆 context),行是 LLM 友好的窗口单位。

#### M3 的取舍

M3 采用 **Cline 风格的三层闸门 + Claude Code 风格的 `cat -n` 行号输出**:

| 层 | 触发条件 | 动作 |
| --- | --- | --- |
| **L1 文件大小** | `stat.size > 20 MB` | 直接拒读(基本上是误读日志/二进制了) |
| **L2 行数** | 切出来 > `limit`(默认 1500)行 | 按行截断,尾部标注总行数 + `start_line=N+1` 续读提示 |
| **L3 字节** | 最终输出 > 400 KB | 按字节截断,尾部标注 |

L2 通常先于 L3 生效(1500 行常常远比 400 KB 早触顶);L3 是兜底,只在某行特别长(minified JS / 单行 5 万字符的 SVG)时挡住。

#### 二进制兜底

LLM 偶尔会把 `.elf` / `.bin` / `.hex` 当文本读,几秒钟就把 context 打满乱码。M3 用一招零依赖的小招拦下:**`readFile` 前先 peek 头 512 字节,扫到 `\0`(NUL 字节)就判二进制并拒绝**。够拦住 90% 的常见误读;真要更严格(UTF-16、CJK 编码嗅探),M4 / M5 再补。

> ⚠️ **唯一漏网之鱼**:Intel HEX(`.hex`)、`.map`、`.lst` 等编译产物是**纯 ASCII 文本**,头 512 字节不会有 NUL,嗅探挡不住。这种只能靠 prompt 教 LLM 别读(附录 B 有对应排查项)。

#### 接口预览

```ts
interface ReadFileInput {
  path: string;         // 必填
  start_line?: number;  // 1-based,默认 1
  limit?: number;       // 默认 1500,上限 5000
}
```

输出格式(`cat -n` + 元数据头 + 截断尾):

```
[文件 vendor/.../boards.txt 共 9821 行,展示 1-1500 行]

     1	# Generic STM32F103C8 (Blue Pill)
     2	# ...
   ...
  1500	GenF1.menu.upload_method...

…(已截断;还剩 8321 行。下一段请用 start_line=1501 续读,或改用 grep 类工具精确定位)

来源:
- vendor/arduino-core-stm32/boards.txt:1-1500
```

三个关键点:

1. **头部一行元数据**(总行数 + 当前展示范围)—— LLM 第一时间知道文件多大、自己看的是哪段,不需要二次猜测。
2. **行号 + tab 前缀**(同 Claude Code) —— 不污染后续 diff;tab 字符让"行号到哪结束"对 LLM 来说无歧义。
3. **截断尾部明写"下一步"** —— 这是 Cline / Claude Code 都验证过的关键策略:模型看到提示后,99% 的概率会乖乖发起 `start_line=N+1` 的续读,而不是放弃或瞎补。

> 🔑 **这一节是 M3 与"业界水平"差距最大的地方**。前面的概念 1~4 是脚手架,代码本身只是这些设计的翻译。如果 `read_file` 还停留在"整读 + 200 KB 截断",`stm32f103xb.h` 800 KB 你怎么读 `PC13` 那段定义?**分段 + 行号 + 续读提示**是让 read_file 真正可用于嵌入式资料的核心。

### 概念 6:PDF 不是文本——为什么要单独一个 `read_pdf`

`read_file` 用 `fs.readFile('.../foo.pdf', 'utf-8')` 读 PDF 会得到一堆 `%PDF-1.4...` 的二进制字节当字符串,**LLM 完全看不懂**。PDF 是结构化文档格式,文本被分散在内部的对象/流里,需要**专门的解析库**抽取。

我们用 **`pdf-parse`** ——一个轻量 npm 包,API 简单:

```ts
import pdf from 'pdf-parse';
import { readFile } from 'node:fs/promises';

const buffer = await readFile('schematic.pdf');
const data = await pdf(buffer);
console.log(data.text);     // 全部页拼接的纯文本
console.log(data.numpages); // 总页数
console.log(data.info);     // 元数据(作者、标题等,经常是空)
```

为什么不把 PDF 解析塞到 `read_file` 里、按扩展名分流?

1. **两个工具的"心智契约"不同**:`read_file` 承诺"读什么得什么",`read_pdf` 内部做了大量启发式抽取(段落顺序、表格、图注),结果**可能与原 PDF 排版不完全一致**。把这两件事分开,LLM 看到 `read_pdf` 就知道"这是抽取过的、可能有损"。
2. **pdf-parse 体积不小(~1 MB)**:不是每个对话都需要它,逻辑分开能让"工具被加载到 LLM 视野里"更精确(以后可以做"按需注册")。

> ⚠️ **pdf-parse 局限**:它只抽**文本**,**对扫描件 PDF 完全无效**(那种是图像,要 OCR)。你给的 `STM32F103C8T6-Blue_Pill.pdf` 是矢量原理图,有文本层,能抽出来——但抽出来的"PA0 / PC13"等是**乱序**的(原理图按视觉位置排,不是阅读顺序)。所以**第一次测试别期望 LLM 通过 PDF 100% 还原原理图**,这就是为什么我们建议**也写一份 `board-bluepill.yaml`** 当备份入口(步骤 8)。

### 概念 7:来源引用(citations)——`ToolResult.sources` 才是 M3 的灵魂

回顾 M0:`shared/index.ts` 当初为什么定义 `Source` 和 `ToolResult<T>`?

```ts
export interface Source { file: string; lines?: string; section?: string; }
export interface ToolResult<T = unknown> { result: T; sources?: Source[]; }
```

当时埋的伏笔是「**输出契约**」(CLAUDE.md):**Knowledge 工具返回必须带来源引用,LLM 答复用户时引用来源,便于核验**。M3 是这个伏笔的第一次兑现。

`read_file` 的 handler 应该这么返回:

```ts
return {
  result: text,                                       // 实际文本内容
  sources: [{ file: 'vendor/.../boards.txt', lines: '3168-3176' }],  // 来源标记
};
```

M2 的 `loop.ts` 里有一段 `stringifyResult` 函数(回看 M2 步骤 8),它会**自动在工具结果末尾追加来源行**:

```
(原始 text 内容)

来源:
- vendor/arduino-core-stm32/boards.txt:3168-3176
```

这段附加文本会随结果一起回填给 LLM,LLM 据此**在最终回答里复述来源**。效果:用户最终看到的 markdown 答复里会写"根据 `vendor/arduino-core-stm32/boards.txt:3168-3176`,Blue Pill 的 LED 接在 PC13",**可核验、不黑箱**。

> 🔑 **如果 M3 只能做对一件事,就是把 `sources` 用好**。这是把 LLM 从"看起来很懂的鹦鹉"变成"可审计的助手"的关键一步,对应 CLAUDE.md 第一条原则的下半句——"output contract"。

### 概念 8:RAG / 向量数据库——是什么、M3 为什么不上、什么时候才上

**简短版**:**RAG 是一种"当资料多到塞不下 prompt 时"的优化,M3 用不到。**

详细解释。LLM 一次对话有**上下文长度上限**(Claude Opus 4.7 是 1M token ≈ 750 万汉字),且**按 token 计费**。当你的资料从「~2 MB 文本」涨到「~几十 MB / 几千页 PDF / 几百颗芯片」,有两个问题:

1. **塞不下**:把全部资料塞进 prompt 超过上限。
2. **算不起 / 慢**:就算塞得下,绝大部分内容跟当前问题无关,白烧 token。

这时上 **R**etrieval **A**ugmented **G**eneration:

| 概念 | 一句话 |
| --- | --- |
| 向量数据库 | 一个能按"**语义相近**"搜索的索引(普通数据库只能"精确匹配") |
| Retrieval | 用户问问题时,先去向量库**搜出最相关的几段**(top-K) |
| Augmented | 把这几段**拼到 prompt** 里 |
| Generation | LLM 根据这几段答题 |

形象类比:

- **塞 prompt 法** = 抱整本字典进考场
- **RAG** = 老师先给你**索引卡片**,考场上翻卡找页码,只看那几页
- **工具调用法**(M3 走这条) = 考场上**你自己决定查什么、看什么**,老师不预先筛选——**LLM 主动调 `read_file`,而不是被动接受系统塞进来的片段**

工具调用 vs RAG 是**两种正交策略**,实际产品里常常**结合**用(LLM 调一个 `search_chip_doc(query)` 工具,工具内部走 RAG)。但**只有"塞 prompt 法明显不够用"时才值得引入 RAG 那套复杂度**。

**M3 不上 RAG 的判据**:

| 信号 | 现状 |
| --- | --- |
| 资料 > 几十 MB,大部分 token 浪费在不相关内容 | ❌ `vendor/` 才 1.85 MB |
| 想支持几百颗芯片 / 几十种板,每对话只用一小撮 | ❌ 现在 1 颗芯片 + 1 块板 |
| token 账单已经心疼 | ❌ 还没大规模用 |

→ **三条都不满足,RAG 是过度工程**。M3 就用最朴素的 `read_file` + 工具循环,**让 LLM 自己决定看什么**(因为它读了 `description` 和 `list_files` 结果后,完全能判断该看哪个文件)。

什么时候才该上?预计 **M5 / M6**,做法是:

- 离线:用 `sqlite-vec` 或 `LanceDB`,把 vendor 数据切片 + 向量化,存本地 `.db` 文件
- 在线:agent 调 `search_chip_doc("PA0 TIM2_CH1 配置")` 这种**新的工具**(对 LLM 来说仍然是工具调用),返回 top-K 片段

→ **从 M3 直接进化到 M5 没有破坏**:就是再多加几个工具进 registry 而已。所以 M3 这一步不会"白做"。

### 步骤 2 · 路径安全 helper(★ 最关键的一段代码)

把概念 3 落成代码。这个文件是 M3 所有文件 I/O 工具的守门员。

**文件:`packages/agent-core/src/tools/paths.ts`**

```typescript
import * as path from 'node:path';

// 工具配置:extension 会在注册时填好
export interface FsToolConfig {
  // 允许访问的根目录(绝对路径)。任何工具读到的最终路径必须落在其中之一下面。
  allowedRoots: string[];
  // 相对路径的解析基准(通常是 workspace 根)。
  // 用户传 'vendor/foo.txt' 时,以它为基准拼成绝对路径再校验。
  workspaceRoot: string;
}

/**
 * 解析用户提供的路径,并校验它落在允许的根目录范围内。
 *
 * - 相对路径 → 相对 workspaceRoot 解析
 * - 绝对路径 → 直接用
 * - 自动处理 ../ 等穿越(path.resolve 会规范化)
 * - 最终绝对路径必须**严格**位于某个 allowedRoot 下,否则抛 PathOutOfRangeError
 *
 * 注意:必须用 `absRoot + path.sep` 做前缀比较,
 *      否则 D:\proj\vendor 会误匹配 D:\proj\vendor2(概念 3 的细节 2)。
 */
export function resolveSafe(inputPath: string, cfg: FsToolConfig): string {
  if (!inputPath || typeof inputPath !== 'string') {
    throw new PathOutOfRangeError('路径参数不能为空');
  }
  const resolved = path.resolve(cfg.workspaceRoot, inputPath);
  for (const root of cfg.allowedRoots) {
    const absRoot = path.resolve(root);
    if (resolved === absRoot || resolved.startsWith(absRoot + path.sep)) {
      return resolved;
    }
  }
  throw new PathOutOfRangeError(
    `路径不在允许范围内:${inputPath}\n允许的根:${cfg.allowedRoots.join(', ')}`,
  );
}

// 自定义错误类型:loop 里可以按它判断"是安全拦截"还是"真出错"
export class PathOutOfRangeError extends Error {
  constructor(message: string) {
    super(message);
    this.name = 'PathOutOfRangeError';
  }
}

/**
 * 把绝对路径转成相对 workspaceRoot 的显示路径(给来源引用用)。
 * 例:D:\MyCode\Embed-agent\vendor\foo.c → vendor/foo.c
 * 跨平台:用 / 而不是 \(LLM 看着更自然,也方便用户在终端复制)
 */
export function toDisplayPath(absPath: string, workspaceRoot: string): string {
  const rel = path.relative(workspaceRoot, absPath);
  return rel.split(path.sep).join('/');
}
```

> 🔑 重读概念 3 那条 `startsWith(absRoot + path.sep)`:**绝不能省 `+ path.sep`**。这是路径校验里被复现最多的 CVE 模式(包括过去 Node 自己也踩过)。

### 步骤 3 · `read_file` 工具

**文件:`packages/agent-core/src/tools/read_file.ts`**

```typescript
import { readFile, stat, open } from 'node:fs/promises';
import type { ToolSpec } from '../types';
import { resolveSafe, toDisplayPath, PathOutOfRangeError, type FsToolConfig } from './paths';

// 三层闸门 + 二进制嗅探;详见概念 5。
const MAX_FILE_SIZE = 20 * 1024 * 1024; // L1:文件大小硬上限 20 MB
const MAX_BYTES = 400 * 1024;           // L3:输出字节兜底 400 KB
const MAX_LINES_DEFAULT = 1500;         // L2:默认行窗口
const MAX_LINES_HARD = 5000;            // limit 参数硬上限,防 LLM 瞎传 9999999
const BINARY_PEEK_BYTES = 512;          // 二进制嗅探:头 N 字节有 NUL 即拒

export function createReadFileTool(cfg: FsToolConfig): ToolSpec {
  return {
    name: 'read_file',
    description: `读取工作区或允许范围内文件的纯文本内容,带行号、支持分段。

用途:查看源码、配置、文档(.md / .yaml / .json / .c / .h / .ts 等)。
参数:
  - path(必填):相对工作区根的路径(如 "vendor/arduino-core-stm32/boards.txt"),
                也可以是绝对路径,但必须落在允许的根目录内。
  - start_line(可选,默认 1):从第几行开始读,**1-based**。
  - limit(可选,默认 ${MAX_LINES_DEFAULT},硬上限 ${MAX_LINES_HARD}):最多读多少行。

返回格式:
  - 头部一行元数据:[文件 <相对路径> 共 N 行,展示 a-b 行]
  - 之后是 cat -n 风格的内容:每行以"行号 + tab"开头(行号是元数据,**不属于文件本体**)。
  - 行数超过 limit 或字节超过 ${Math.floor(MAX_BYTES / 1024)} KB 时,
    末尾会标注「…(已截断,下一段请用 start_line=N 续读)」。

注意:
  - 大文件(> 50 KB)请优先用 start_line + limit 分段读,不要整读。
  - 本工具不读 PDF / 二进制文件(.elf / .bin / .hex);请改用 read_pdf 或先 list_files 查看扩展名。
  - 单文件超过 ${MAX_FILE_SIZE / 1024 / 1024} MB 会被直接拒读。`,
    inputSchema: {
      type: 'object',
      properties: {
        path: { type: 'string', description: '文件路径(相对工作区根,或允许范围内的绝对路径)' },
        start_line: { type: 'number', description: '起始行号(1-based,默认 1)' },
        limit: {
          type: 'number',
          description: `最多读多少行(默认 ${MAX_LINES_DEFAULT},上限 ${MAX_LINES_HARD})`,
        },
      },
      required: ['path'],
      additionalProperties: false,
    },
    handler: async (input) => {
      const { path: p, start_line, limit } = input as {
        path: string;
        start_line?: number;
        limit?: number;
      };

      // ① 路径安全校验(所有 fs 工具的统一入口)
      let abs: string;
      try {
        abs = resolveSafe(p, cfg);
      } catch (e) {
        if (e instanceof PathOutOfRangeError) {
          // 把"拒绝"作为正常的工具结果回填,让 LLM 据此向用户解释,
          // 而不是抛异常导致 loop 直接错误退出。
          return { result: `拒绝:${e.message}` };
        }
        throw e;
      }

      // ② 类型检查 + L1 文件大小硬闸门
      const st = await stat(abs);
      if (st.isDirectory()) {
        return { result: `拒绝:${p} 是目录,请改用 list_files。` };
      }
      if (st.size > MAX_FILE_SIZE) {
        return {
          result:
            `拒绝:${p} 大小 ${(st.size / 1024 / 1024).toFixed(1)} MB,` +
            `超过单文件 ${MAX_FILE_SIZE / 1024 / 1024} MB 限制。\n` +
            `建议:用 list_files 探明、grep 类工具(M4 起)精确定位,或自行拆分文件后再读。`,
        };
      }

      // ③ 二进制嗅探:peek 头 512 字节,扫到 NUL 即判二进制
      const fd = await open(abs, 'r');
      try {
        const peek = Buffer.alloc(BINARY_PEEK_BYTES);
        const { bytesRead } = await fd.read(peek, 0, BINARY_PEEK_BYTES, 0);
        if (peek.subarray(0, bytesRead).includes(0)) {
          return {
            result:
              `拒绝:${p} 看起来是二进制文件(头 ${bytesRead} 字节含 NUL)。\n` +
              `PDF 请用 read_pdf;其它二进制本工具不支持。`,
          };
        }
      } finally {
        await fd.close();
      }

      // ④ 参数归一化:start_line ≥ 1、limit 夹在 [1, MAX_LINES_HARD]
      const startLine = Math.max(1, Math.floor(start_line ?? 1));
      const lineLimit = Math.min(
        MAX_LINES_HARD,
        Math.max(1, Math.floor(limit ?? MAX_LINES_DEFAULT)),
      );

      // ⑤ 读全文(L1 已限 ≤ 20 MB,内存可控)
      const full = await readFile(abs, 'utf-8');
      const allLines = full.split('\n');
      // POSIX 文件多以 \n 结尾,split 会多出一个空 element;pop 掉,行数与 cat -n 一致
      if (allLines.length > 0 && allLines[allLines.length - 1] === '') {
        allLines.pop();
      }
      const totalLines = allLines.length;
      const displayPath = toDisplayPath(abs, cfg.workspaceRoot);

      // ⑥ 空文件特例
      if (totalLines === 0) {
        return {
          result: `[文件 ${displayPath} 是空文件]`,
          sources: [{ file: displayPath }],
        };
      }

      // ⑦ L2 行窗口切片
      const startIdx = startLine - 1;
      if (startIdx >= totalLines) {
        return {
          result: `拒绝:start_line=${startLine} 超过文件总行数 ${totalLines}。`,
        };
      }
      const endIdxExclusive = Math.min(totalLines, startIdx + lineLimit);
      const lineSlice = allLines.slice(startIdx, endIdxExclusive);
      const lastLineNum = startIdx + lineSlice.length;
      const lineTruncated = endIdxExclusive < totalLines;

      // ⑧ 格式化:cat -n 风格(行号 padStart 到 6 宽 + tab)
      let body = lineSlice
        .map((line, i) => `${String(startIdx + i + 1).padStart(6)}\t${line}`)
        .join('\n');

      // ⑨ L3 字节闸门(防一行特别长的 minified JS / SVG 撑爆)
      let byteTruncated = false;
      if (Buffer.byteLength(body, 'utf-8') > MAX_BYTES) {
        body = body.slice(0, MAX_BYTES);
        byteTruncated = true;
      }

      // ⑩ 组装输出:元数据头 + 内容 + 「下一步」截断尾(概念 7 的 sources 同步带出范围)
      const header =
        `[文件 ${displayPath} 共 ${totalLines} 行,` +
        `展示 ${startLine}-${lastLineNum} 行]\n\n`;
      let footer = '';
      if (lineTruncated) {
        const remaining = totalLines - lastLineNum;
        footer =
          `\n\n…(已截断;还剩 ${remaining} 行。` +
          `下一段请用 start_line=${lastLineNum + 1} 续读,或改用 grep 类工具精确定位)`;
      } else if (byteTruncated) {
        footer =
          `\n\n…(已截断:输出超过 ${Math.floor(MAX_BYTES / 1024)} KB,` +
          `请缩小 limit 或用 grep 类工具定位)`;
      }

      return {
        result: header + body + footer,
        sources: [{ file: displayPath, lines: `${startLine}-${lastLineNum}` }],
      };
    },
  };
}
```

> 看几个细节(请逐条读懂,这段比 M2 的 demo 工具长不少,但每一段都对应概念 5 的某条约束):
>
> - **`handler` 第一行立刻调 `resolveSafe`**——所有路径校验都走它,没有第二条入口。
> - **拒绝不抛异常,而是返回 `拒绝:…` 文本**——LLM 能继续作答(告诉用户为什么被拒),否则 `loop.ts` 会把异常变成"工具执行失败"把会话推向错误流。目录、超大文件、二进制、`start_line` 越界都遵循相同模式。
> - **三层闸门的执行顺序**:`stat` 判文件大小(L1)→ peek 嗅二进制 → 读全文 → 行窗口切片(L2)→ 字节兜底(L3)。L2 通常先于 L3 生效(1500 行 ≈ 50 KB,远比 400 KB 早触顶);L3 只在某行特别长(minified JS / 单行巨大 SVG)时挡住。
> - **截断尾部明写"下一步"**:`start_line=N+1` 续读 / grep 类工具——这是让 LLM **主动续读**而不是放弃的关键策略,业界(Claude Code、Cline、Codex)都验证过。
> - **`padStart(6)` 把行号补到 6 字符宽**:`     1` ~ `999999`,对齐美观;tab 让"行号到哪结束"对 LLM 无歧义。
> - **二进制嗅探只看头 512 字节**:足以拦下 90% 的常见误读(`.elf` / `.o` / `.bin` 文件头几十字节一定有 NUL);但 Intel HEX(`.hex`,全 ASCII)、`.map` 等编译产物会漏过去——交给 prompt 教 LLM 别读(附录 B 有排查项)。
> - **`sources.lines`** 写成 `"3000-3099"`,这样 M2 步骤 8 的 `stringifyResult` 拼"来源:"段时会自动带出具体行范围,**用户在最终答复里能看到「板子接 PC13(来源:board.yaml:3000-3099)」**,而不是只看到文件名。

### 步骤 4 · `list_files` 工具

**文件:`packages/agent-core/src/tools/list_files.ts`**

```typescript
import { readdir, stat } from 'node:fs/promises';
import * as path from 'node:path';
import type { ToolSpec } from '../types';
import { resolveSafe, toDisplayPath, PathOutOfRangeError, type FsToolConfig } from './paths';

const MAX_ENTRIES = 200; // 单次列目录返回上限,防止把整个 node_modules 倒出来

export function createListFilesTool(cfg: FsToolConfig): ToolSpec {
  return {
    name: 'list_files',
    description: `列出工作区或允许范围内某个目录的文件与子目录(不递归)。

用途:在读文件之前先探明目录里有什么、文件是不是 PDF、有没有同名的 .h/.c 配对等。
参数:
  - path(可选):要列出的目录路径(相对工作区根,或允许范围内的绝对路径);
                 不填则列出工作区根目录。
返回:每行一条,形如:
  - "[D] subdir/"   表示子目录
  - "[F] file.txt   1.2 KB"   表示文件 + 大小
最多返回 ${MAX_ENTRIES} 条;超出会标注「…(已截断)」。`,
    inputSchema: {
      type: 'object',
      properties: {
        path: {
          type: 'string',
          description: '目录路径,默认 "."(工作区根)',
        },
      },
      additionalProperties: false,
    },
    handler: async (input) => {
      const { path: p = '.' } = (input as { path?: string }) ?? {};

      let abs: string;
      try {
        abs = resolveSafe(p, cfg);
      } catch (e) {
        if (e instanceof PathOutOfRangeError) {
          return { result: `拒绝:${e.message}` };
        }
        throw e;
      }

      const st = await stat(abs);
      if (!st.isDirectory()) {
        return { result: `拒绝:${p} 不是目录,请改用 read_file。` };
      }

      const entries = await readdir(abs, { withFileTypes: true });
      // 按目录在前、然后字母序——给 LLM 看更整齐
      entries.sort((a, b) => {
        if (a.isDirectory() !== b.isDirectory()) return a.isDirectory() ? -1 : 1;
        return a.name.localeCompare(b.name);
      });

      const lines: string[] = [];
      let count = 0;
      for (const e of entries) {
        if (count >= MAX_ENTRIES) {
          lines.push(`…(已截断,共 ${entries.length} 条,只显示前 ${MAX_ENTRIES})`);
          break;
        }
        if (e.isDirectory()) {
          lines.push(`[D] ${e.name}/`);
        } else {
          const size = (await stat(path.join(abs, e.name))).size;
          lines.push(`[F] ${e.name}   ${formatSize(size)}`);
        }
        count++;
      }

      return {
        result: lines.length > 0 ? lines.join('\n') : '(空目录)',
        sources: [{ file: toDisplayPath(abs, cfg.workspaceRoot) + '/' }],
      };
    },
  };
}

function formatSize(n: number): string {
  if (n < 1024) return `${n} B`;
  if (n < 1024 * 1024) return `${(n / 1024).toFixed(1)} KB`;
  return `${(n / 1024 / 1024).toFixed(1)} MB`;
}
```

> `readdir(..., { withFileTypes: true })` 是 Node 比较新的写法,一次拿到 entry 类型,**不用对每个名字再 stat 一次去判**目录还是文件——效率高。我们对**文件**仍单独 stat 取大小,这是 Node API 的限制(`Dirent` 不带 size)。

```typescript
import { readFile } from 'node:fs/promises';
import pdf from 'pdf-parse';
import type { ToolSpec } from '../types';
import { resolveSafe, toDisplayPath, PathOutOfRangeError, type FsToolConfig } from './paths';

const MAX_CHARS = 30000; // PDF 抽出来的文本可能很长,粗暴截断

export function createReadPdfTool(cfg: FsToolConfig): ToolSpec {
  return {
    name: 'read_pdf',
    description: `读取 PDF 文件的文本内容(抽取纯文本,丢失版面)。

用途:读芯片手册、板子原理图、规格书等 PDF 资料。
参数:
  - path(必填):PDF 文件路径(相对工作区根,或允许范围内的绝对路径)。
返回:抽取的纯文本 + 总页数;超过 ${MAX_CHARS} 字会被截断。
注意:
  - 仅适用于"有文本层"的 PDF;扫描件(图像 PDF)抽出来会是空字符串。
  - 复杂版面(原理图、表格)的抽取结果**顺序可能错乱**——
    如果发现内容像被打乱,优先尝试用户工程里的结构化资料(如 board.yaml),
    或建议用户提供更清晰的来源。`,
    inputSchema: {
      type: 'object',
      properties: {
        path: {
          type: 'string',
          description: 'PDF 文件路径',
        },
      },
      required: ['path'],
      additionalProperties: false,
    },
    handler: async (input) => {
      const { path: p } = input as { path: string };

      let abs: string;
      try {
        abs = resolveSafe(p, cfg);
      } catch (e) {
        if (e instanceof PathOutOfRangeError) {
          return { result: `拒绝:${e.message}` };
        }
        throw e;
      }

      const buf = await readFile(abs);
      const data = await pdf(buf);

      let text = data.text ?? '';
      let truncated = false;
      if (text.length > MAX_CHARS) {
        text = text.slice(0, MAX_CHARS);
        truncated = true;
      }

      const header = `[PDF 共 ${data.numpages} 页,${truncated ? '已截断' : '完整抽取'}]\n\n`;
      return {
        result: header + text + (truncated ? '\n\n…(已截断)' : ''),
        sources: [{ file: toDisplayPath(abs, cfg.workspaceRoot) }],
      };
    },
  };
}
```


```typescript
  // workspace 根:用户当前打开的工程(可能没打开,fallback 到 extensionUri 同级)
  const workspaceFolder = vscode.workspace.workspaceFolders?.[0]?.uri.fsPath;
  const workspaceRoot = workspaceFolder ?? this.context.extensionUri.fsPath;

  // 允许读的根:workspace 自己 + 工程下两个常见调研目录(若存在)
  const allowedRoots = [
    workspaceRoot,
    path.join(workspaceRoot, 'vendor'),
    path.join(workspaceRoot, 'MeterialsForResearch'),
  ];
  return { workspaceRoot, allowedRoots };
}

const fsCfg = buildFsConfig();
const registry = new ToolRegistry();
for (const t of demoTools) registry.register(t); // M2 演示工具可保留
registry.register(createReadFileTool(fsCfg));    // ★ M3 新增
registry.register(createListFilesTool(fsCfg));   // ★
registry.register(createReadPdfTool(fsCfg));     // ★
```

几个要点:

- **`workspaceRoot` 用 VS Code API 拿**——这就是为什么这段代码必须在 `extension` 包(agent-core 不能 `import 'vscode'`,M2 概念 8)。
- **`allowedRoots` 包含 `workspace` 自己**:这样 LLM 也能读你写的 `board-bluepill.yaml`、源码、`doc/` 下文档。
- **`vendor` / `MeterialsForResearch` 是否存在不影响**:`fs.readFile` 在路径不存在时会自然抛 ENOENT,handler 里的 `stat` 调用会兜住——但即使我们提前 fs.exists 也只是优化,不是必需(因为 `resolveSafe` 只校验**字符串**,不要求实际存在)。

| --- | --- | --- |

## 附录 C · 名词表

- **path traversal(路径穿越)**:利用 `../` 跳出预期目录的攻击,经典 Web 漏洞。
- **allow-list / 白名单**:只允许列表里的东西,其余一律拒绝(对应 deny-list 黑名单)。
- **citation / 来源引用**:答案里附上"我这话从哪里来"的标记,M3 通过 `ToolResult.sources` 实现。
- **factory(工厂函数)**:不直接 `export` 实例,而是 `export` 一个造实例的函数;M3 三个新工具都是工厂。
- **CJS / ESM**:CommonJS(`require/module.exports`,老)与 ES Module(`import/export`,新)。`pdf-parse` 是 CJS,要用 default import 接。
- **RAG(检索增强生成)**:M3 不上,概念 8 解释了为什么。
- **reasoning_content / thinking 模式**:DeepSeek thinking / R1 等模型在流里**额外**吐一段"思考过程"文本(独立于正文 `content`)。多轮对话(含工具循环)里必须把上一轮的 `reasoning_content` 原样回传给 API,否则 400。详见**附录 D**。
- **`<details>` / `<summary>`**:HTML5 原生折叠组件,浏览器自带切换 + 键盘 + ARIA;`<details open>` 表示默认展开。M3 用它给"思考过程"做零 JS 折叠区。

## 附录 D · DeepSeek thinking 模式兼容 + 思考过程 UI

### D.1 症状

接好 M3 工具后,如果在设置面板里选了 DeepSeek 的**思考型模型**(如 `deepseek-chat-thinking` 等任何带 thinking 行为的模型),首次让 agent 调任何工具(`list_files`、`read_file`…)时会立刻报:

```
请求失败:400 The `reasoning_content` in the thinking mode must be passed back to the API.
```

并且工具循环直接终止,聊天面板不再继续回答。

> ⚠️ Anthropic(claude-opus-4-7 / sonnet-4-6)和 OpenAI 普通模型(gpt-4o 等)**不会**触发这条错误——它们的"思考"机制不同(Anthropic 走 `thinking` 块,OpenAI 内部不暴露),不会强求把思考回传。

### D.2 根因

DeepSeek 的思考型模型流式响应包含**两条独立字段**:

| 字段 | 内容 | 当前我们怎么处理 |
| --- | --- | --- |
| `delta.content` | 正式回答正文(你在聊天里看到的) | ✅ 累加进 assistant 气泡 |
| `delta.reasoning_content` | 思考过程文本(模型"出声想") | ❌ M3 初版完全没读 |

DeepSeek 把 `content` 和 `reasoning_content` 当作**模型上一轮发言不可分割的两半**。在多轮请求(工具循环里"调完工具再问模型续答"是典型场景)的 assistant 消息里,如果只有 `content` 没有 `reasoning_content`,API 认为"你篡改了我的发言"→ 直接 400。

工具调用恰好是"高频多轮场景"——所以 M3 一开就崩。

### D.3 修复设计

**最小修复**只关心一件事:**在流里捕获 `reasoning_content` → 累加进我们的 `ChatTurn` → 下一轮请求里原样回传**。不需要展示给用户也能修。

**增强 UI** 是 M3 的福利:既然已经收到这段思考,顺手在聊天面板里用**可折叠的"💭 思考过程"区**展示出来,用户可以选择看 / 不看。

### D.4 改动文件清单

| 文件 | 改动 |
| --- | --- |
| `packages/agent-core/src/types.ts` | `LlmStreamEvent` 加 `\| { type: 'reasoning'; text: string }`;`ChatTurn` 的 assistant 变体加可选 `reasoningContent?: string`;`AgentEvent` 也加 `reasoning` 分支供 extension 转发 |
| `packages/agent-core/src/adapters/openai.ts` | 流里读 `(choice.delta as { reasoning_content?: string }).reasoning_content` 并 `yield { type: 'reasoning', text }`;`toOpenAiMessages` 把 `turn.reasoningContent` 塞回 assistant 消息的 `reasoning_content` 字段 |
| `packages/agent-core/src/loop.ts` | 用 `let assistantReasoning = ''` 累加,push history 时 `...(assistantReasoning ? { reasoningContent: assistantReasoning } : {})`,并把 `reasoning` 事件原样转发给上层 |
| `packages/agent-core/src/loop.test.ts` | 加一条用例钉死"reasoning 累加进 history + 转发给上层"的行为 |
| `packages/shared/src/index.ts` | `ExtToWebview` 加 `\| { type: 'reasoningDelta'; text: string }`(extension → webview 的新消息类型) |
| `packages/extension/src/ChatViewProvider.ts` | 收到 `reasoning` AgentEvent:同时 post `reasoningDelta` 给前端 + `output.append` 打到 Output 面板 |
| `packages/webview/src/store.ts` | `ChatItem` 的 assistant 变体加 `reasoning?: string`;新增 `case 'reasoningDelta'`,规则同 `streamDelta` 但追加到 `reasoning` 字段;思考先于正文到达时,新建一个 `text: ''`、`reasoning: msg.text` 的 assistant 气泡 |
| `packages/webview/src/components/MessageItem.tsx` | 在主气泡上方渲染 `<details open><summary>💭 思考过程</summary>...</details>`;reasoning 用 `whitespace-pre-wrap` + 左侧 2px 边条 + 斜体降级显示 |

### D.5 几个关键决策

1. **`reasoning` 单独成事件,不复用 `text`**——避免上层混淆"模型正式说的话"和"模型自言自语",前者要进 markdown 渲染,后者保留原貌。
2. **用 HTML5 `<details>/<summary>` 而不是 React 状态**——零状态、零事件监听,浏览器自带键盘 + ARIA + 折叠动画。默认 `open`(展开)。
3. **思考先于正文到达时,先建 `text: ''` 的 assistant 气泡**——store 的"最后一项是 assistant 就追加"规则不变,但 `text` 为空时主气泡 div 不渲染(`{(isUser || item.text) && ...}`),UI 干净。
4. **Output 面板同时打日志**——即便用户折叠了思考区,开发者(你)开"输出 → Embed Agent"频道仍能看到完整思考流,debug 用。
5. **Anthropic adapter 暂不动**——它有自己的 `thinking` 块机制,要支持的话改的是 `anthropic.ts` 而非 `openai.ts`,且 Anthropic 不会因为缺 thinking 块报硬错。留到真要用 Anthropic 思考模型时再加。

### D.6 给 TS 新手的几个新模式

| 模式 | 哪里出现 | 解读 |
| --- | --- | --- |
| `(choice.delta as { reasoning_content?: string }).reasoning_content` | `openai.ts` | OpenAI SDK 官方类型里没声明这个字段(因为是 DeepSeek 扩展的),用窄类型 as 断言"我知道这字段可能存在"。比 `as any` 更精确——只放行这一个字段。 |
| `out.push({ role: 'assistant', ..., ...reasoningPart } as ChatCompletionMessageParam)` | `openai.ts` | 同上,我们想塞进一个 SDK 类型不认的字段。先正常构造对象,最后 `as` 一次到外层类型——比对每个字段 cast 干净。 |
| `...(condition ? { foo: bar } : {})` | `loop.ts` | "可选字段"惯用法:`if condition` 就展开 `{foo: bar}`,否则展开空对象。结果对象不会带 `undefined` 字段,比 `{ foo: condition ? bar : undefined }` 干净。 |
| `item.role === 'assistant' ? item.reasoning : undefined` | `MessageItem.tsx` | TS 三元的真值分支里 item 被收窄成 assistant 变体——只在那一支才能访问 `item.reasoning`(user 变体没有这字段)。 |
| `<details open>` + `<summary>` | `MessageItem.tsx` | HTML5 原生折叠;`open` 是属性而非 prop,React 写 `<details open>` 等价于 `open={true}`。 |

### D.7 思考区的细节行为(已实现)

加完基础的"可折叠思考区"之后,又做了两项 UX 细节:

- **限高 + 流式自动滚到底**:思考 div 加 `max-h-64 overflow-y-auto`(16rem ≈ 256px,~12 行);每次 `reasoning` 变化用 `ref.current.scrollTop = scrollHeight` 把滚动条贴底,保证用户能跟上正在生成的思路。代价:用户手动滚回去看上面的话,新片段会把他们拽回底部——折叠思考区再展开能避开。
- **正文一到 500ms 后自动折叠**:首段 `item.text` 到达后等 500ms,程序化 `setOpen(false)`。用户的注意力转移到正式答案上,思考区淡出。**实现要点**:
  - `<details>` 用 `open={open}` 受控,加 `onToggle` 把用户的手动展开/折叠**同步回 state**——否则 React 下次渲染会强制覆盖,用户"点不动"。
  - `useRef` 拍下挂载瞬间的 `hasText`(`initialHasTextRef`)。挂载时正文已在(刷新 webview / 历史回放)→ 初始就是折叠 + 自动折叠闸门关闭,不再去打扰用户。
  - `autoCollapsedRef` 一次性闸门:自动折叠只发生一次,之后无论用户怎么开关都不再覆盖。

### D.8 还没做的(留给后续)

- **Anthropic thinking 块兼容**:Anthropic API 的 `thinking` 是 message 里的 block,不是流式 delta 的单独字段。如果以后要用 Claude opus 4.7 的 extended thinking,需要在 `anthropic.ts` 里加类似但不同的处理。
- **token 计量拆分**:DeepSeek 的 reasoning tokens 会算钱(单价低)。当前只显示总用量,没拆"思考 token"和"正文 token"。如果用户对账单敏感,后续在 `usage` 事件里加细分字段。
