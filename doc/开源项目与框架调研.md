## 〇、结论速览(TL;DR)

把项目拆成几个能力维度,各维度「最值得看 / 最值得用」的答案:

| 维度 | 现状(本项目) | 最值得**借鉴**的开源项目 | 最值得**直接用**的框架/库 | 建议动作 |
| --- | --- | --- | --- | --- |
| **LLM 多 provider + 流式 + 工具循环** | M2 手写 thin adapter(Anthropic/OpenAI/DeepSeek) | Cline、Continue | **Vercel AI SDK**(`ai`,Apache‑2.0) | 手写版先跑通(有教学价值);达到触发条件后切 AI SDK,见 §二 |
| **可插拔工具框架** | 自写 `ToolRegistry` + `ToolSpec` | Cline / Continue 的 MCP 接入 | **MCP TypeScript SDK**(`@modelcontextprotocol/sdk`) | 现在保留自写 registry,但**预留 MCP 适配位**,见 §四 |
| **VS Code 插件 + Webview 架构** | React18+Zustand+Tailwind 自定义 webview | **Continue**(core/gui/extension 三层最像你) | VS Code 原生 Chat/LM API(替代方案,§五) | 维持自定义 webview(契合「配置即对话」愿景);读 Continue 架构 |
| **Diff-first 写文件 + 确认** | 计划中(护栏已定) | **Cline**(diff+checkpoint+approval)、**Aider**(edit format / repo map) | VS Code 原生 diff viewer | Phase 3 解冻前先抄 Cline 的 approval/diff 交互 |
| **架构模式(循环/压缩/权限/缓存)** | M2 雏形 | **Claude Code / Cline / Codex / OpenCode** 架构拆解 | — | §七 列了 8 条现在就能吸收的模式 |
| **嵌入式数据/工具(冻结)** | 数据源已分析(Zephyr 为主) | zephyr‑dts、Zephyr IDE、modm‑devices | `kconfiglib`(子进程)、`dtc` 兜底 | 解冻后再启用,§八 |
| **Webview UI 组件** | 自己用 Tailwind + VS Code 主题变量 | — | (官方 `@vscode/webview-ui-toolkit` **已废弃**,你的选择正确) | 维持现状;若要组件库看 VSCode Elements |

**三个最重要的判断**:

1. **你正在 M2 手写的那层 adapter,是整个项目里"最该被现成框架替换"的部分。** [Vercel AI SDK](https://github.com/vercel/ai) 做的就是这件事(统一 Anthropic/OpenAI/DeepSeek + 流式 + 多步工具循环),Apache‑2.0、TS 原生、生态第一。手写版作为**学习**很好,但长期维护多 provider 的边角(流式拼 JSON、各家工具格式差异、重试)会很累——这些 AI SDK 已经替你趟平。详见 §二。
2. **"可插拔工具框架"的事实标准是 MCP**(Model Context Protocol)。你自写的 `ToolSpec` registry 没问题,但应**从一开始就让它能接 MCP**,这样将来任何 MCP server(文件系统、数据库、甚至别人写的嵌入式工具)都能即插即用。详见 §四。
3. **不必重新发明 agent 的"骨架知识"。** Cline / Continue / Claude Code 这些头部项目把"工具循环、权限、上下文压缩、diff 应用"的坑都踩过了,且多为 Apache‑2.0,可大方借鉴(§三、§七)。

> 📌 针对"**Cline 是否自研 / 要不要上 LangChain·LangGraph / 嵌入式怎么落到现有架构**"的专题深入,见 **§十一**。

## 一、调研范围与方法

**为什么做这个调研**:`CLAUDE.md` 强调「技术栈已定型,勿重新选型」,但当前 M2 正在**手写 LLM adapter + 工具循环**——这恰好是开源生态里"轮子最多、最成熟"的一块。在投入大量自写代码前,值得看清:哪些能直接复用、哪些值得借鉴思路、哪些坚持自研更好。

**调研维度**(对应项目分层):

- Presentation:VS Code 插件 + Webview(同类 AI 编码插件怎么搭)
- Agent Core:多 provider 适配、流式、工具调度、确认/权限、上下文管理(框架 vs 自写)
- Tool Layer:工具如何"可插拔"(MCP 等协议)
- Data Layer(冻结):嵌入式硬件数据 / DTS / Kconfig 的现成解析库与参考工具

**方法**:检索 2025–2026 的一手资料(GitHub 仓库、官方文档、架构拆解长文),优先看**许可证友好(Apache/MIT/BSD)、TS 生态、与本项目架构同构**的项目。所有结论附来源链接(§十一)。

## 二、最关键的决策:Agent 编排层——继续手写 adapter,还是用框架?

这是本次调研**最该认真对待的一个决定**,因为它正是你此刻(M2)在写的代码。

### 2.1 现状回顾

M2 文档里你的方案是:自写一层 `LlmAdapter` 接口,下面挂 `AnthropicAdapter` / `OpenAiAdapter`(DeepSeek 复用 OpenAI + `baseURL`),自己用 `async function*` 处理流式、自己拼工具调用的流式 JSON、自己写 `loop.ts` 做多步工具循环。**思路完全正确**——这正是适配器模式的教科书用法。问题只是:**这件事已经有人做成了高质量、Apache‑2.0、TS 原生的库。**

### 2.2 候选框架横向对比

| 框架 | 定位 | 多 provider(含 DeepSeek) | 流式 | 内置工具循环 | 重量 | 许可 | 适配本项目? |
| --- | --- | --- | --- | --- | --- | --- | --- |
| **手写 thin adapter**(现状) | 自研薄封装 | 自己实现 | 自己实现 | 自己实现(`loop.ts`) | 极轻 | 你的(Apache) | 学习价值高;长期维护成本上升 |
| **Vercel AI SDK**(`ai`) | 轻量、流式优先的 TS 工具包 | ✅ 一等公民(`@ai-sdk/anthropic`/`openai`/`deepseek`) | ✅ 原生 | ✅ `streamText`+`tools`+`stopWhen` 自动多步 | 轻 | **Apache‑2.0** | ⭐⭐⭐⭐⭐ **最贴合** |
| **Mastra** | 全功能 TS agent 框架(agent+workflow+memory+RAG+eval+Studio) | ✅ 统一 router(~94 家) | ✅ | ✅ | 中重 | 开源(以仓库为准) | ⭐⭐⭐ 现在偏重;Phase 2+(RAG/记忆/评测)再看 |
| **LangGraph.js / LangChain.js** | 状态图编排,人在环、检查点、时间旅行调试 | ✅ | ✅ | ✅(graph) | 重、学习曲线陡 | MIT | ⭐⭐ 复杂多 agent 才值;现在过度设计 |
| **Claude Agent SDK**(TS) | Anthropic 出品,自带 agent loop+工具+MCP+子 agent | 偏 Claude 生态 | ✅ | ✅ | 中 | 以仓库为准 | ⭐⭐ 与"多 provider 含 DeepSeek"诉求冲突 |
| **OpenAI Agents SDK**(TS) | 轻量多 agent handoff,内置 tracing | 偏 OpenAI 生态 | ✅ | ✅ | 轻 | 以仓库为准 | ⭐⭐ 同上,provider 中立性不足 |

### 2.3 逐个点评

**① Vercel AI SDK(`ai`)——最贴合本项目的选项**

- 它的 `streamText` + `tool()` + `stopWhen` 几乎**一比一覆盖你 M2 的 adapter + loop**:统一接口、流式、自动多步工具循环、`abortSignal` 取消、`usage` 用量回传——全是你正在手写的东西。
- **provider 中立**正是你要的:换厂商只改一个 model 工厂,`@ai-sdk/anthropic` / `@ai-sdk/openai` / `@ai-sdk/deepseek`(DeepSeek 有官方 provider,无需再靠 `baseURL` 凑)。
- **Apache‑2.0**,与本项目许可一致,无 GPL 传染风险;TS 生态装机量第一,边角 case(各家流式事件差异、工具参数流式拼接、429/5xx 重试)都帮你处理了。
- 跑在 Node 端,esbuild 打包无障碍(和现有 `extension.js` 一致)。

直观对比——你 M2 里 `anthropic.ts` + `openai.ts` + `loop.ts` 的核心,用 AI SDK 大约是这样十几行:

```ts
import { streamText, tool, stepCountIs } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';   // 换 provider 只改这一行
import { z } from 'zod';

const result = streamText({
  model: anthropic('claude-opus-4-7'),
  system: SYSTEM_PROMPT,
  messages,
  tools: {
    get_current_time: tool({
      description: '返回当前时间',
      inputSchema: z.object({}),
      execute: async () => new Date().toISOString(),
    }),
  },
  stopWhen: stepCountIs(10),   // 多步工具循环上限(替代你手写的 while)
  abortSignal: signal,         // 取消(替代你手写的 AbortController 传递)
});

for await (const part of result.fullStream) {
  // part.type: 'text-delta' | 'tool-call' | 'tool-result' | 'finish' …
  // 直接映射成你的 ExtToWebview 事件
}
```

> 你的**确认原语(`requiresConfirm`)依然是你自己的**——AI SDK 不管 VS Code 弹窗。做法:对需确认的工具**不写 `execute`**,AI SDK 会把 tool‑call 交还给你,你弹确认卡片、用户点头后再把结果回填(这正是 AI SDK 官方的"human‑in‑the‑loop"模式,与你的 confirm 设计天然契合)。

**② Mastra——更"重"的全功能框架**

- 不只做对话,还含 **workflow(可暂停/人审/分支并行)、memory、RAG、eval、可视化 Studio**。这些恰好对应你**冻结阶段**的需求(嵌入式知识 RAG、评测体系 M4、记忆)。
- 现在(M2 通用底座)用它**偏重**;但当 Phase 2+ 要做 RAG + 评测 + 多步工作流时,值得回头评估。许可证以仓库为准,使用前确认。

**③ LangGraph.js / LangChain.js——状态图,强但重**

- 强项是**复杂、长程、可恢复(检查点)、人在环、多 agent**的编排,还能"时间旅行调试"。MIT。
- 但学习曲线陡、Python 为主 TS 为辅、代码量大。对你"一个聊天 + 工具循环"的底座是**过度设计**。记在备选,等真的需要复杂图状态再说。

**④ Claude Agent SDK / ⑤ OpenAI Agents SDK——provider 中立性不足**

- 两者都很好,但都**偏自家模型生态**。你的硬需求是"多 provider、用户自带 key、含 DeepSeek",用任一家的 agent SDK 都会和这点别扭。可借鉴其**设计**(尤其 Claude Agent SDK 的工具/hooks/子 agent、MCP 深度),但不建议作为底座框架。

### 2.4 建议(对 M2 的具体处理)

**尊重现计划的教学价值,但留好退路**:

1. **M2 现在**:按 `M2-Agent对话核心-详细文档.md` 手写 adapter 跑通——对"会 Vue、TS/React 入门"的你,亲手实现一遍工具循环/流式/适配器,**理解深度**是框架给不了的(这与 milestone 文档"手把手教学"的定位一致)。
2. **把 `LlmAdapter` 接口守干净**(你已经这么设计了):`loop.ts` 只认 `LlmAdapter`,不碰具体 SDK。这意味着将来**整层换成 AI SDK 只动 adapter 文件**,`loop`/`registry`/extension 接线都不用改。这是你现有设计最大的红利,务必保持。
3. **触发"切到 Vercel AI SDK"的条件**(满足任一即认真评估):
   - 要接**第 3 个以上** provider,或被各家流式/工具格式差异反复绊住;
   - 需要**结构化输出**(`generateObject`)、或想接 **MCP 工具**(AI SDK 原生支持 MCP tools);
   - 手写 adapter 的 bug 开始吃掉本该用于产品功能的时间。
4. 一句话总结(也是头部 agent 架构拆解的共识):**"AI 调用本身是次要的,差异化在 harness(编排/安全/上下文)"**——所以**把可替换的 provider 层尽量交给成熟库,把精力留给你独有的部分**(VS Code 集成、confirm/diff 护栏、配置即对话的 UI、嵌入式知识)。

## 三、可借鉴的同类开源项目(VS Code AI Agent 参考实现)

这些是和你**最像**的产品——VS Code 里的 AI 编码 agent。它们把"工具循环 + 权限 + diff + MCP"在真实产品里跑通了,且多为 Apache‑2.0,读源码/借思路都安全。

| 项目 | 语言/许可 | 与本项目的相似点 | 最值得借鉴什么 | 注意 |
| --- | --- | --- | --- | --- |
| **Cline** | TS / Apache‑2.0 | VS Code 扩展 + 工具 + 审批 + diff;5M+ 安装 | 工具审批流、diff+checkpoint(可撤销)、MCP 接入、Plan/Act 模式、`.clinerules` | 功能大而全,挑着看 |
| **Continue.dev** | TS / Apache‑2.0 | **架构和你几乎同构**:`core` + `gui`(React)+ `extension`,三者**消息传递** | 三层职责划分、`config.yaml` 模型角色(chat/edit/apply/embed/rerank)、上下文 provider 机制 | 跨 VS Code/JetBrains/Neovim,有抽象成本 |
| **Kilo Code** | TS / Apache‑2.0 | Cline+Roo 的合并 fork,活跃维护 | Orchestrator 模式(拆子任务路由到专用模式)、内联补全 | 较新,功能多 |
| **Roo Code** | TS / Apache‑2.0 | 模式化:Architect/Code/Debug/Ask/Custom | "模式 = 一组工具子集"的思路 | ⚠️ **仓库已于 2026‑05‑15 归档**,只作思路参考,别 fork |
| **Aider** | Python / Apache‑2.0 | CLI 形态的编辑型 agent | **edit format(search/replace、unified diff)**、**repo map**(tree‑sitter 提取签名给模型,不塞全文)、自动 git 提交 | Python,不能直接复用代码,但**思路对你的 diff‑first 和上下文压缩极有价值** |
| **Void** | TS / Apache‑2.0 | 开源版 Cursor | 整体 UX、provider 设置 | 是 **VS Code 整体 fork**(不是扩展),架构差异大,远观即可 |

**逐个划重点**:

- **Cline(首选精读)**:与你目标形态最接近。重点看它的 **(a) 工具审批 / 权限交互**(每步 diff/命令/浏览器动作都要用户点头——正是你 `requiresConfirm` + diff‑first 的成熟版);**(b) checkpoint/撤销**(每次编辑打快照,可回退);**(c) MCP 集成**(任意 MCP server 即插即用);**(d) Plan/Act 切换**(先规划后执行,契合你"禁止自主多步执行后再报告"的护栏)。Apache‑2.0,放心借鉴。

- **Continue(架构对照首选)**:它的仓库结构 `core`(核心逻辑)+ `gui`(React 侧栏 webview)+ `extensions/vscode`(Node 宿主)+ `packages`(共享 npm 包),三者**靠已知协议消息传递**——这和你的 `agent-core` / `webview` / `extension` / `shared` **几乎一一对应**。你在纠结"这段逻辑该放哪一层"时,直接看 Continue 怎么分。它的 `config.yaml` 把模型按**角色**(chat/edit/apply/embed/rerank)拆开,对你将来"开发期 opus / 生产 sonnet / 子任务 haiku"的多模型策略很有参考价值。

- **Aider(借思路不借码)**:虽是 Python CLI,但两件事是金子——**① edit format**:它不让模型整文件重写,而是输出 `SEARCH/REPLACE` 块或 unified diff,token 省、可审查(直接对应你 Phase 3 的 `propose_file_edit` 怎么设计输出格式);**② repo map**:用 tree‑sitter 抽取整库的类/函数签名做一张"地图"喂给模型,而不是塞全文——这是"Tool‑driven, not context‑stuffing"原则的极佳实现,你将来让 agent 理解用户工程时会需要它。

## 四、可插拔工具:Model Context Protocol(MCP)

**这条直接回应你 Phase 1 的核心目标之一——"可插拔工具框架"。**

- **是什么**:MCP 是把"给 LLM 提供工具/资源/提示"标准化的开放协议(Anthropic 发起,现已是事实标准)。一个 MCP **server** 暴露一组 tools/resources/prompts;你的 agent 作为 **client/host** 去 `listTools` / `callTool`。Cline、Continue、Claude、Kilo 全都支持它。
- **现成库**:[`@modelcontextprotocol/sdk`](https://github.com/modelcontextprotocol/typescript-sdk)(官方 TS SDK,Tier‑1,npm 下载量 6600 万+,生态最大)。同时提供 server 端和 client 端(stdio / Streamable HTTP 等传输)。
- **为什么对你重要**:你现在的 `ToolRegistry` + `ToolSpec` 是**第一方工具**的好设计;但如果**只**有它,所有工具都得你自己写。接上 MCP 后:文件系统、数据库、Git、甚至将来别人写的"嵌入式工具",都能作为外部 MCP server **零代码接入**。这把"可插拔"从"我能加工具"升级成"整个生态的工具都能即插即用"。

**建议(不打乱 M2)**:

1. **M2/M3 维持自写 registry**——它简单、可测、够用,先把第一方工具(demo、`read_file` 等)跑通。
2. 但**给 registry 预留一个"MCP 适配位"**:即写一个未来的 `registerMcpServer(client)`,把 MCP server 的 `listTools()` 结果转成你的 `ToolSpec` 注册进同一张表。这样**核心循环一行不改**,就能在某个里程碑接通 MCP(你的 `ToolSpec` 已经用 JSON Schema 描述参数,与 MCP 工具 schema 同构,转换成本很低)。
3. 若 §二 采纳了 Vercel AI SDK,则更省事——**AI SDK 原生支持把 MCP tools 直接喂给 `streamText`**。

## 五、要不要用 VS Code 原生 AI API?

VS Code 自己提供了一套 AI 扩展 API,这是一个**架构层面的岔路口**,值得明确知道你"为什么没走这条路"。

| 原生 API | 作用 | 对你的取舍 |
| --- | --- | --- |
| **Chat Participant API** | 注册 `@embed-agent` 聊天参与者,复用 VS Code **原生聊天 UI** | 省掉自写 webview;但**让出 UI 控制权** |
| **Language Model API** | 通过用户的 Copilot 订阅访问模型 | 省掉 provider 适配;但**走 Copilot,难自带 DeepSeek key、provider 受限** |
| **Language Model Tool API** | 让模型在原生 orchestration 里调用你注册的工具 | 工具能力现成;但绑定原生编排 |
| `@vscode/vscode-chat-extension-utils` | 官方"建聊天参与者"辅助库 | 配合上面用 |

**结论:你选自定义 webview 是对的,但要知道代价。**

- **为什么自定义 webview 合理**:你的产品愿景是「**配置即对话** + 可视化配置 UI + diff/审计层 + 引脚表等」(规划 §一、Phase 4)。原生 Chat UI 给不了这种**深度定制的可视化**;而且原生 LM API 走 Copilot,和你"**多 provider、用户自带 key、含 DeepSeek、私有网关**"的硬需求冲突。
- **你让出的东西**:原生 Chat UI 的开箱体验、与编辑器更紧的集成、自动跟随 VS Code 演进。这些你要自己补。
- **务实建议**:维持自定义 webview 作主路径。可选——**将来加一个轻量 Chat Participant 作"入口快捷方式"**(用户在原生 Chat 里 `@embed-agent` 也能唤起你的 agent),两条入口并存。不急,记在 backlog。

## 六、其它"现成轮子"(配套库)

按需取用,多数是对你**已定型技术栈**的补强或验证:

- **Webview UI 组件**:官方 **`@vscode/webview-ui-toolkit` 已于 2025‑01 废弃/归档**(因底层 FAST 项目转向)——所以你**没用它、改用 React+Tailwind+VS Code 主题变量是正确的**。若将来想要"原生观感"的组件,社区替代是 **VSCode Elements**(基于 Lit)、VSCode Community UI Toolkit。但你现在的方案够用,不必动。
- **Schema 校验 / 工具入参**:你工具参数用 JSON Schema(对的,两家 SDK 都吃这个)。若引入 Vercel AI SDK 或想要运行时校验,**Zod** 是 TS 生态标配(`z.object(...)` 既当 schema 又当运行时校验,还能反推类型),可与 JSON Schema 互转。
- **Token 计数**:`CLAUDE.md` 提到 tiktoken。注意 tiktoken 在 Node 里常需 WASM/原生件;若想要**纯 JS、零原生依赖**,可看 `gpt-tokenizer`。不过——**两家 API 响应里已带真实 usage**(你 M2 也是这么取的),所以**短期内根本用不上本地分词器**,等真要"发请求前预估成本"再加。
- **本地向量库(冻结阶段)**:`CLAUDE.md` 选了 **sqlite-vec**(轻、嵌入式友好、跟着 sqlite 走)——是稳妥选择。备选 **LanceDB**(列式、面向更大规模与多模态)。Phase 2 解冻做 RAG 时再定。
- **可观测/评测(M4 及以后)**:若用 Mastra 自带 eval;否则自写 harness 即可(你 M4 计划已是这思路)。重模型追踪可看 LangSmith / OpenLLMetry,但现在不必。

## 七、可直接吸收的架构模式(来自头部 agent 拆解)

下面这些来自对 **Claude Code / Cline / Codex / OpenCode** 的架构拆解,是"别人用血踩出来的坑"。**标 ✅ 的现在(M2/M3)就能低成本吸收**,其余记进 backlog:

1. ✅ **工具在注册时声明元数据**(`isReadOnly` / `isConcurrencySafe` / 是否需确认)。你已有 `requiresConfirm`;**再加一个 `readOnly` 标志**几乎零成本,将来可据此让只读工具**并行执行**(头部 agent 只读工具并发可达 10),写工具串行防冲突。
2. ✅ **权限拒绝当作"对话反馈"回填给模型**。用户拒绝某工具时,不要静默失败,而是把"用户拒绝了 X"作为 tool_result 喂回去,模型会自己调整方案。**你 M2 的 `loop.ts` 已经这么做了**(拒绝/失败也 push 一条结果)——这点做得很对,保持。
3. ✅ **流式里"边流边显示工具进度"**。你 M2 已有 `toolCallStart/Result`,继续保持;头部 agent 更进一步做**流式中就开始执行只读工具**(streaming tool execution)降低延迟——记进 backlog。
4. ✅ **system prompt 拆成可组合的小节**(身份/规则/风格/环境/工具指引),而非一大坨字符串。便于 A/B 与维护。你现在 prompt 很短,等它变长时按此拆。
5. **缓存友好的 prompt 装配**:把"静态行为指令"和"会话相关内容"用边界分开,静态部分可命中 provider 的 prompt cache,省钱。Anthropic 的 prompt caching 尤其值得用——等你 prompt/工具清单变大再上。
6. **多策略上下文压缩**:长会话必然撞上下文上限。头部 agent 有"主动压缩 / 撞 `prompt_too_long` 后被动压缩 / 截断"多重兜底。你 M2 先做了"留最近 N 轮 + 单工具结果截断"(对的,别过早做摘要);把"撞错再压缩"记进 backlog。
7. **diff/编辑有多种格式**(search‑replace / unified diff / 行范围编辑),各有取舍。Phase 3 设计 `propose_file_edit` 时,优先 **search/replace 或 unified diff**(token 省、可审查,参考 Aider/Codex)。
8. **文件状态缓存**(LRU 缓存读过的文件,避免重复读、支持撤销)。你做 `read_file`(M3)和将来写文件时用得上。

> 一句话:**产品的护城河不在"调用了 LLM",而在这层 harness**(编排、安全、上下文、diff)。这正是你 `CLAUDE.md` 四条设计原则(tool‑driven / diff‑first / read‑free‑write‑gated / 版本绑定)的外部印证——继续沿着它走。

## 十、给本项目的落地建议(分阶段)

**现在 / Phase 1(M2–M4),低风险、立即可做:**

1. **M2 按现计划手写 adapter 跑通**(教学价值),但**严守 `LlmAdapter` 接口边界**,使其将来可被整体替换。
2. **吸收 §七 标 ✅ 的架构模式**:给 `ToolSpec` 加 `readOnly` 标志;保持"拒绝/失败也回填结果";prompt 变长时拆小节。这些几乎零成本。
3. **精读两个项目**:**Continue**(对照你的分层与消息协议)、**Cline**(对照工具审批 / diff / MCP)。各花半天读源码,省你大量试错。
4. **给工具 registry 预留 MCP 适配位**(一个将来要实现的 `registerMcpServer`),现在只留口子、不实现。

**中期 / 触发条件满足时(见 §2.4):**

5. **评估切换到 Vercel AI SDK**:接第 3 个 provider、需要结构化输出/MCP 工具、或手写 adapter 维护成本上升时,用 AI SDK 替换 `adapters/*`(`loop`/`registry`/extension 接线不动)。
6. **接通 MCP**(`@modelcontextprotocol/sdk` 或经 AI SDK):让外部工具即插即用。
7. **Diff‑first 落地参考 Cline 的 approval+checkpoint、Aider 的 edit format / repo map。**

**解冻后 / Phase 2+(嵌入式):**

8. **DTS 解析先用 `zephyr-dts`**,不够再自研;Kconfig 走 `kconfiglib` 子进程(已定)。
9. **RAG/评测/记忆**成为主线时,回头评估 **Mastra**(全家桶)或在 AI SDK 上自拼。
10. 嵌入式工具(查芯片/生成代码)作为 **MCP server 或 `ToolSpec`** 接入,**底座不改**——这正是你架构一开始就追求的"可插拔"红利兑现。

## 十一、深入专题:Cline 是否自研 · LangChain/LangGraph 取舍 · 嵌入式落地预想(2026‑05 追加)

> 本节回应一次具体追问:**Cline 到底用没用框架?本项目要不要上 LangChain/LangGraph?嵌入式阶段会怎么落到现有架构?** 是 §二(框架取舍)、§三(同类项目)、§八(嵌入式)的深入,结论可直接执行。

### 11.1 Cline 用框架了吗?——没有,全自研,且收敛到"原生工具调用"

Cline(5M+ 安装,与本项目最像)**没有用 LangChain/LangGraph,也没用 Vercel AI SDK**。它把整个 agent 运行时自己写了一遍,2026 年还专门推倒重建成可复用的 `@cline/sdk`。三块核心全自研,且**与你 agent-core 的形态几乎一一对应**:

| 组件 | Cline 的做法 | 对应你 agent-core |
| --- | --- | --- |
| Provider 抽象 | `@cline/llms`:一个 `ApiHandler` 接口,挂各家 handler(Anthropic/OpenAI/Gemini/Bedrock/Mistral/vLLM…),**直接调官方 SDK**,不经框架 | `LlmAdapter` + `adapters/*` |
| Agent 循环 | 自写,"无状态、可复用",自管流式/上下文/错误 | `loop.ts` 的 `runAgent` |
| 工具系统 + MCP | 自写 `createTool` 注册 + 原生 MCP 接入 | `ToolRegistry` + `ToolSpec`(MCP 待接) |

**一个对你是强验证的细节**:Cline 早期用"在 system prompt 里描述工具、让模型吐 XML、再自己解析"的土办法,多工具序列成功率只有 70–80%、常报 `invalid API response`;2026 年 v3.35 **迁移到原生工具调用**(工具以 JSON Schema 直接发给 API,模型回原生 JSON tool call),成功率升到接近 100%,省 ~15% token,还能并行执行工具。**而你的 M2 从第一天就用原生工具调用**(Anthropic `tool_use` / OpenAI function calling)——Cline 迭代很久、踩坑后才到的终点,你起步即站对。

> **小结**:与你最像、最成功的产品**刻意不用框架**,而是手写薄 provider 抽象 + 循环 + 工具系统——这正是你 agent-core 的形态。Roo / Kilo / Continue / Aider 的核心同样不用 LangChain。这本身就是对"要不要上框架"最有力的回答。

### 11.2 本项目要不要上 LangChain / LangGraph?

两者是不同的东西,分开判:

**LangChain(大杂烩工具包)——不建议。**

- 你需要的部分(provider 抽象、agent 循环、工具系统)**已落地并有离线单测**;
- 剩下可能想要的(Phase 2 RAG 的 loader/splitter/向量封装),你已选更轻的替代(`yaml` + 自切块 + `sqlite-vec` + embedding),为省几十行胶水引入大依赖、还要吃它频繁的破坏性升级与 Python 优先设计,对 TS 新手团队不划算;
- 它会**接管你最该自控的编排层**,而编排/安全/上下文才是护城河。

**LangGraph(状态图编排)——现在不要;Phase 3 再评估,且未必是它。**

- 它的强项是复杂、长程、可中断恢复(checkpoint)、时间旅行调试、多 agent 图;对你"单 agent + 工具循环"的底座是过度设计,其 human‑in‑the‑loop interrupt 还与你**已和 webview 打通的确认原语**功能重叠(你的更简单)。
- **硬伤**:LangGraph 的 **TS 是二等公民**(Python 优先,TS 文档/示例少),对纯 TS 栈 + TS 入门者很不友好。
- **唯一值得回头看的点**:Phase 3「生成配置 → 编译 → 读报错 → 修 → 重编」若演化成需要**确定性、有次数上限、可恢复**的状态机。但即便那时:diff‑first/确认/写入护栏始终是你自己的;这个修复循环用**现有 loop + 一个重试计数器 + `get_last_build_log`** 通常就够;真要上工作流框架,**Mastra(TS 原生,含 workflow/暂停/人审/分支)比 LangGraph 更适合你的团队**,优先级在其之上。

> **一句话裁决**:**不用 LangChain;现在不用 LangGraph。** 保留自研 agent-core(与 Cline 一致、被验证的正确路线)。LangGraph 放进"Phase 3 codegen 工作流真复杂时再评估、且先看 Mastra"的候选名单。唯一现在值得认真考虑的框架仍是 **Vercel AI SDK**——但它替换的是 provider 层、不是编排,且不急(见 §二)。

### 11.3 嵌入式后续开发预想

**最关键判断:嵌入式能力 = 往现有底座"加工具",不是重写。** 你 M2 写好的 loop / registry / adapter / confirm 是**永久底盘**,后面几乎不动——这是 §三、§四、§十 反复强调的"可插拔"红利的兑现。

**能力如何落到现有架构:**

| 阶段 | 新增能力 | 落在哪 | 动底座? | 现成东西 |
| --- | --- | --- | --- | --- |
| 入口 | `get_project_meta`:读 `west.yml`/board/Zephyr 版本,识别工程 → 选对应版本索引 | 新 `ToolSpec` | 否 | `simple-git` |
| Phase 2 知识(只读) | `lookup_binding`/`lookup_kconfig`/`list_boards`/`search_zephyr_docs`/`search_zephyr_samples` | 新 `ToolSpec` + `hw-data` 包 + 离线索引 | 否 | `zephyr-dts`、`yaml`、`kconfiglib`(子进程)、`sqlite-vec` |
| Phase 3 配置+代码生成(写) | `propose_file_edit`/`propose_multi_file_change`(事务)、`run_west`(gated)、`get_last_build_log`、`validate_config` | 新 `ToolSpec` + diff/确认流程(extension 层) | 底座不变;extension 加 diff 流程 | VS Code diff API;编辑格式借 Aider/Codex |
| Phase 4 可视化 | board 面板、引脚表、配置 inspector | webview 新组件 ↔ agent 事件双向同步 | 否 | — |

> 你 M0 就定的 `ToolResult<T>` + `Source[]` 输出契约,正是给 Phase 2 知识工具"带来源引用"准备的——现在看是先见之明。

**三个具体前瞻建议:**

1. **Python 工具的桥:先子进程,长大后升级成 MCP server。** `kconfiglib`/`dtc`/`west` 是 Python/原生的,TS agent-core 调不动。起步(Phase 2 MVP)在 `ToolSpec` 的 handler 里 `spawn` 子进程(`CLAUDE.md` 已定 kconfiglib 走 Python 子进程);长大后把整个 Zephyr 知识层封成**独立 MCP server**(可用 Python 写,kconfiglib/dtc 原生在 Python),TS agent-core 作 MCP client 接入——进程隔离(重解析/索引不拖累激活预算,对上风险表)、语言原生、可独立测试、可复用。这正兑现 §四"给 registry 预留 MCP 适配位"的伏笔。
2. **RAG 保持轻,别为它上 LangChain。** bindings 查询用**精确查找**(key 是 `compatible`,字典/索引即可,比向量准);docs/samples 才用**向量 RAG**(`sqlite-vec`)。索引**按 Zephyr release tag 隔离**(版本绑定)。这套规划里已有,无需重框架。
3. **杀手锏是"生成前校验",这是你数据策略的兑现点。** CubeMX 数据无 schema,而 **Zephyr 的 bindings YAML 就是配置格式 schema**。所以你能在出 diff 前,拿 bindings + Kconfig **校验 agent 生成的配置**,自动拦经典坑(`CONFIG_FOO=y` 写成 `"y"`、漏 `pinctrl-names`、overlay 放错位、HWM v1/v2 差异)。做成 `validate_config` 工具接进循环,生成质量会拉开差距。

**编排会不会复杂到需要框架?——不会,复杂度在别处。** "配芯片 + 生成代码 + 编译修复"看似复杂,但复杂度集中在**工具内部**(DTS 解析、索引、校验、代码模板)和**知识/prompt**里,**编排始终是那个简单的循环**。框架解决的是"编排复杂度",而你的复杂度不在那儿——这正是 tool‑driven 原则的意义,也是手写 loop 能一直撑住的原因。
